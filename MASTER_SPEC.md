## 1. 本文件解决什么问题

这个项目现在最容易出问题的地方，不是“代码写不出来”，而是：

- 规则真源不唯一
- client / server / shared 的边界经常漂移
- README、GDD、docs/agent、实际代码状态容易不同步
- 测试主要覆盖协议链路，但不总能覆盖可见交互与游戏反馈
- 多 Agent 协作时，经常顺手扩 scope，导致反复 Debug

因此，从这一版开始，仓库只保留两份核心产品文档：

1. `MASTER_SPEC.md`：定义目标世界和边界
2. `WORK_QUEUE.md`：定义执行顺序和任务卡

除此之外：
- `README.md` 只负责项目介绍
- `docs/agent/*` 只负责状态记录、基线审计和历史记忆
- `docs/archive/*` 与历史 GDD 只作为参考，不再覆盖本文件

## 2. 对当前技术路线的判断

### 2.1 结论

当前 Demo 1 **不应立刻换语言、换引擎、重写仓库**。

保留当前方向：
- client：TypeScript + Phaser 3 + Vite
- server：Node.js + Express + Socket.IO
- shared：共享类型、协议、静态数据

### 2.2 原因

这个仓库的目标是：
- Web-first
- 2D 俯视角 / 2.5D 视觉伪装
- 房间制联机
- 验证“搜 -> 打 -> 撤”的短局闭环

对这个目标来说，现有技术路线是成立的。当前的高成本问题更像是：
- 规格治理不稳
- 契约冻结不足
- 可见反馈不足
- 多 Agent 协作缺少强约束

### 2.3 什么时候才考虑迁移

只有满足下面任一条件，才在 Demo 1 之后评估 Godot / Defold / 其他工具：

- 目标变成移动原生优先，而非 Web-first
- 项目开始高度依赖可视化编辑器、碰撞编辑、关卡编辑器
- Web 包装移动端已明显阻碍交付
- 团队不再接受“浏览器优先”的内容生产流程

在达到这些条件之前，**治理优先于重写**。

## 3. Demo 1 的唯一目标

Demo 1 是一个 **Web-first、房间制、多人联机、2D 俯视角搜打撤垂直切片**。

唯一要验证的是：

> 玩家能否在同一局地图中完成“搜资源 -> 打怪 / 对抗 -> 背包取舍 -> 前往撤离点 -> 撤离或失败结算”的完整闭环。

---

## 4. Demo 1 范围

## 4.1 必做

- 创建房间 / 加入房间
- 多玩家进入同一局
- 玩家移动与基础战斗
- 至少三把武器具备明确差异
- 怪物（普通 / 精英）
- 掉落
- 背包与装备系统
- 玩家死亡掉落
- 固定撤离点
- 成功 / 失败结算
- 结算后返回大厅，可继续下一局

## 4.2 不做

以下全部不进入 Demo 1：

- 交易行
- 保险箱
- 打造 / 强化
- 复杂匹配
- 复杂账号体系
- 聊天系统
- 世界 Boss
- 撤离道具
- 自由技能搭配
- 商业化后端

只要任务触碰到上面内容，默认视为越界。

## 5. 当前仓库的目标架构

## 5.1 三层边界

### client/
负责：
- Lobby
- GameScene
- 实体表现
- HUD / 背包 UI / 结算 UI
- 输入与反馈
- Socket 客户端接入

禁止：
- 最终命中判定
- 最终掉落判定
- 最终死亡 / 撤离 / 结算判定

### server/
负责：
- 房间与对局生命周期
- 玩家状态权威
- 怪物状态权威
- 战斗判定
- 掉落生成
- 拾取校验
- 死亡 / 撤离 / 结算

### shared/
负责：
- 类型定义
- 协议事件名
- 共享枚举
- 静态数据
- 全局运行参数

规则：
**任何影响 client 与 server 一致性的变更，都必须先落到 shared。**

## 6. 以后所有 Agent 的硬规则

## 6.1 文档优先级

固定优先级：

1. `MASTER_SPEC.md`
2. `WORK_QUEUE.md`
3. `docs/agent/CANONICAL_BASELINE.md`
4. `docs/agent/DELTA_MATRIX.md`
5. `docs/agent/DECISIONS.md`
6. `docs/agent/PROJECT_STATE.md`
7. 代码当前行为
8. `README.md`
9. GDD / `docs/archive/*` / 旧分析 / 口头描述

如果内容冲突：
- 先列冲突
- 不准直接猜
- 不准一边写代码一边改规格
- 先更新任务卡，再决定是否改代码

## 6.2 契约规则

只要涉及以下内容，必须先改 shared：
- Socket 事件名
- payload 字段
- 共享枚举
- 背包结构
- 武器 / 怪物 / 地图共享参数

## 6.3 范围规则

每次只允许做一张任务卡。
禁止：
- 一轮同时做规格重写 + 逻辑重构 + 视觉美化
- 顺手加功能
- 顺手改相邻系统
- 没有列文件范围就直接改代码

## 6.4 记录规则

每次任务结束必须更新：
- `docs/agent/PROJECT_STATE.md`
- `docs/agent/OPEN_LOOPS.md`
- `docs/agent/WORKLOG.md`

## 7. 这次代码审查后确认的主要风险方向

> 这里写的是“优先验证并收口的风险方向”，不是让 Agent 立刻大改全仓。

### 风险 A：真源过多

README、GDD、docs/agent 和记忆文档过多时，Agent 很容易选错“当前世界观”。

### 风险 B：shared 契约消费方式可能漂移

本项目必须把 shared 作为唯一契约层；如果 client / server 对 shared 的读取方式不统一，极容易产生“本地看起来改了，运行时还是旧行为”的幽灵 Bug。

### 风险 C：背包 / 掉落 / 装备的“数据模型”和“UI 表达”不一定完全同构

这会导致：
- 服务端是一个世界
- 测试看到的是另一个世界
- Agent 再继续基于错觉开发

### 风险 D：技能入口、输入入口、HUD 暴露与服务端真实能力可能不同步

### 风险 E：自动化脚本覆盖主链，但可见交互的回归不够强

这类项目如果只有 Socket 主链脚本，没有浏览器端 smoke test，就很容易陷入“协议通了，但试玩还是怪”的状态。

## 8. 本轮不直接写死的参数

从这一版开始，下列运行参数不要靠 README、GDD 或口头描述当真源：

- 地图尺寸
- 撤离开放时间
- 撤离半径 / 读条时间
- 出生规则
- 怪物刷新规则
- 武器参数
- 背包尺寸与占格规则

这些参数的当前真实值，必须由 **Task 01：Baseline Audit** 从实际代码抽取并落到：

- `docs/agent/CANONICAL_BASELINE.md`

在 Baseline 没生成之前：
- 不允许因为“印象里应该是多少”而改代码
- 不允许随意把 README 或 GDD 里的值强行覆盖到实现

## 9. 本地 Agent 的角色分工

### GPT-5.4
负责：
- 总控
- 架构判断
- shared / server / scripts / docs
- 任务拆解

### 人类开发者
负责：
- 最终是否“像游戏”的判断
- 是否允许进入下一阶段
- 是否接受大改与越界改动

规则：
- Gemini 不得擅自改 shared、协议、服务端规则
- GPT-5.4 不得在没有视觉基线和参考图的情况下大面积自由改 UI 风格

## 10. 本轮进入代码前，必须先完成的审计结果

任何 Agent 在第一次真正改代码前，必须先输出：

1. 当前仓库目录与关键入口摘要
2. 当前实现与本文件的差异矩阵
3. 当前最危险的 5~10 个 Debug 根因
4. P0 / P1 / P2 分阶段计划
5. 第一张任务卡
6. 第一张任务卡允许修改的文件
7. 第一张任务卡禁止修改的文件
8. 验收方式

如果上面没输出完整，默认不得开始编码。
