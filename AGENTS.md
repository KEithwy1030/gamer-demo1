# AGENTS.md - 流荒之路开发规则

任何接管本项目的 AI agent 必须遵守本文件。

本项目首先是一款游戏，其次才是软件工程。工程、测试、文档和流程只用于稳定交付玩家价值，不能代替玩家价值。

本文件只记录长期规则，不记录当前任务、进度勾选或阶段完成状态。可玩状态以 Git 提交、经验证的 checkpoint 和实际运行证据为准。

---

## 1. 产品身份

**流荒之路**是一款 8-15 分钟一局的中世纪末世硬核轻量化 PvPvE 搜打撤游戏。

- 核心循环：搜 -> 打 -> 撤 -> 卖。
- 核心情绪：搬砖快感、紧张、贪婪。
- 玩家是依靠装备求生的拾荒者，不是高速操作型英雄。
- 死亡与撤离必须有真实得失，黑市与永久档案承接局外积累。
- 完整产品采用真正的 **2.5D 斜视空间**，不是平面 2D 俯视图叠斜视纸片。

用户已明确推翻 `GDD.md` 中“Demo 1 可以用 2D 假装 2.5D、不强求真 2.5D”的旧例外。该句在修订前属于历史文本，不得作为继续制作平面 2D 的依据。

成功首先用玩家语言判断：玩家能做什么、看到什么、听到什么、感到什么、选择什么、失去或得到什么，以及是否想继续下一局。

以下内容都只是支持证据，不能单独宣布游戏或阶段完成：

- 测试全绿
- 构建通过
- 架构整洁
- 提交、事件、文档或截图数量
- 没有已知 bug
- agent 对自己实现的解释

## 2. 真源与读取顺序

进入项目先确认本地真相：

```powershell
git status --short --branch
git branch --all --verbose --no-abbrev
git log --all --graph --decorate --oneline -n 40
git tag --sort=-creatordate --format="%(creatordate:short) %(refname:short) %(objectname:short) %(contents:subject)"
```

- 只读分析、审计或答疑默认不执行 `git fetch`，除非用户要求远端最新状态，或当前判断确实依赖远端新提交。
- 获得代码/文档写入授权的开发任务，在编辑前确认 remote 与目标 branch，再执行 `git fetch`。fetch 只更新远端引用，不自动 merge/rebase。

随后按任务读取最少必要文档：

- 所有产品开发：读 `docs/QUALITY-BAR.md` 和 `docs/REFACTOR-GUIDE.md` 第一部分。
- 产品定位或玩法取舍：再读 `PITCH.md`。
- 数值、规则或服务端合同：再读 `GDD.md` 的相关章节，并用当前代码核验文档是否陈旧。
- Godot、2.5D 客户端或迁移工作：必须先有并读取 Godot 迁移边界、垂直切片合同、资产生产规范和环境记录；缺失时先建立这些边界，不直接开始全量迁移。
- 资产工作：读取当前美术圣经、资产清单和相关 source/runtime 记录。
- 音频工作：读取音频事件表、总线规划和当前资产清单。

优先级：当前用户指令 > 本文件 > 经确认的产品合同 > 当前代码与运行证据 > 旧状态文档和历史说明。

不一次性加载整个项目文档库。历史审计只作为证据，不自动成为当前计划。

## 3. 工作模型：Producer 与独立验收

“非平凡增量”是指成功与否需要判断完整玩家节拍，或会改变玩法状态、存档/档案、联机协议、场景空间、角色动作、可见 UI、资产、VFX 或音频体验的工作。确定的文字、路径、注释、无行为配置修正通常属于小型任务。

非平凡的玩法、场景、角色、UI、资产、音频或可玩片段必须分离职责：

1. **Main Session / Producer-Director**
   - 定义阶段边界、玩家承诺、优先级和验收。
   - 不编辑当前候选的运行时代码、场景、玩法数据、测试或资产。
   - 最终接受、退回、部分关闭或开启下一阶段。
2. **Builder**
   - 是当前增量唯一的产品写入者。
   - 实现并完成局部自检，但不能接受自己的工作。
3. **QA**
   - 只读验证规定路径、状态、回归和工程底线。
   - 不顺手修复，不与 Builder 共同解释失败。
4. **Experience Reviewer**
   - 先把游戏作为黑盒体验，再看实现背景。
   - 评判第一眼、空间、动作、反馈、选择、压力、节奏、视听一致性和继续游玩动机。

同一工作区同一时间只能有一个 Builder 写产品文件。QA、Reviewer 和 Producer 保持只读，避免共享工作区并发修改。

小型低风险任务可以由单个 agent 实现并验证，例如确定的文字、资源路径或局部 P0 修复；但单次自测不能被夸大为体验验收。

如果当前环境无法提供独立 QA 或 Reviewer，Builder 可以继续完成自检和非依赖工作，但对应 Functional、Experience 或 Delta 状态必须标记为 `NEEDS_REVIEW`，不能自行宣布通过。

## 4. 每个阶段必须有 Goal Card

任何非平凡增量开始前，Producer 必须用一屏以内的简短 Goal Card 明确：

- **Segment Name**：阶段名称。
- **Segment Start**：玩家从哪里、什么状态开始。
- **Segment End**：玩家做到什么才结束。
- **Player Promise**：玩家将做、感受、注意、选择或在意什么。
- **Canonical Path**：规定的可重复体验路径。
- **Baseline**：对比的旧版本、提交或录像。
- **In Scope / Out of Scope**：本阶段做什么、不做什么。
- **Engineering Floor**：最低功能与回归保护。
- **Experience Evidence**：需要的录像、连续帧、声音或状态证据。
- **Human Gate**：哪些维度确定需要真人，以及为什么 agent/VM 不能可靠完成。

阶段要小到能形成一个完整玩家节拍：第一眼 -> 玩家行动 -> 世界/角色反馈 -> 后果 -> 过渡。

Goal Card 可以存在于当前任务计划、对话或 checkpoint 信息中，不要求为了流程单独创建文档。已从上游合同明确继承的字段直接引用，不重复抄写。

禁止用“优化体验”“完善游戏”“修复若干问题”作为无边界阶段。

## 5. Creation 优先与缺陷预算

默认工作是玩家可感知的 Creation：

- 可玩空间、路线、场景和内容
- 角色动作、怪物行为和战斗节拍
- 资产、动画、音频、VFX 和世界反馈
- 选择、压力、后果与收益仪式
- 克制而可读的游戏原生 UI

只有确认的 P0/P1 缺陷可以打断当前 Creation：

- **P0**：无法启动、存档/档案损坏、不可逆数据错误、安全问题、主路径完全阻断。
- **P1**：当前玩家承诺无法完成，或核心画面/输入/反馈严重错误到无法评审。
- **P2**：明显问题，但有可用路径且不阻断当前阶段。
- **P3**：轻微瑕疵、边缘兼容、低频或纯装饰问题。

P2/P3 进入 backlog，不因为容易修、容易测试或截图显眼而成为当前主线。

单个 P0/P1 最多采用两种不同根因方案，且默认不超过 Goal Card 预计阶段投入的约 20%。不同方案必须基于不同因果假设或实现路径，不把同一函数反复微调算成新方案。20% 是强制复盘和换路触发点，不是允许带着 P0/P1 接受候选；没有安全 workaround 时，候选保持阻塞或部分关闭。

如果连续两个增量只增加测试、文档、重构、日志、截图、提示文字或小修，下一个增量必须交付玩家可感知 Creation，除非仍有确认的 P0/P1 阻断。

## 6. 完成标准：不是第一目标全绿

第一目标是 Player Promise 真实出现，并比 Baseline 有实质提升。测试用于证明它能稳定存在。

候选依次通过四层：

### 6.1 Engineering Floor

- 相关构建、导入、协议和局部回归通过。
- 无启动崩溃、状态重复结算、档案损坏或确认的 P0/P1。
- 日常增量只跑与改动相关的最小测试集。
- 阶段 checkpoint 跑相关回归包；发布候选才跑完整发布门禁。

不要为了追求全仓测试数字，在每个小改动后运行所有测试或修复无关 P2/P3。

### 6.2 Functional Gate

- 独立 QA 能从 Segment Start 到达 Segment End。
- 输入、状态、后果、存档/档案和必要联机链路真实工作。
- 玩家不依赖开发者口头解释也能执行关键行动。

### 6.3 Experience Gate

黑盒 Reviewer 按 0-5 分评审，并按以下权重换算为百分制：

| 维度 | 权重 |
| --- | ---: |
| 第一眼与游戏世界 | 15 |
| 玩家动作与空间交互 | 15 |
| 反馈与手感 | 15 |
| 选择、压力与后果 | 15 |
| 节奏、情绪与继续动机 | 15 |
| 美术、动画与音频一致性 | 15 |
| UI 克制与可读性 | 10 |

每项得分按 `评分 / 5 * 权重` 计算，总和为百分制结果。

阶段通过基线：加权至少 `75/100`，第一眼、动作、反馈、节奏和视听一致性不得低于 `3/5`。每个分数必须引用具体时间点、画面、动作或声音证据。

### 6.4 Delta Gate

相对 Baseline 至少出现一种可观察提升，且没有显著倒退其它核心维度：

- 新的有意义玩家行动或空间交互
- 新的选择、压力、后果或路线
- 更强的动作、世界响应、声音或控制感
- 增加继续游玩动机的场景、资产或情绪节拍
- 原本被阻断的玩家承诺变得可玩且可理解

代码量、测试数、文档、提交数、截图数和“没有 bug”不属于 Delta。

决定只有：`ACCEPT`、`RETURN`、`PARTIAL`、`REJECT`、`NEEDS_REVIEW`。退回时 Producer 最多给三个决定性的玩家问题。

## 7. 测试环境与主机授权

### 7.1 环境顺序

测试环境按以下优先级选择：

1. **本机后台测试**：默认首选。
2. **隔离虚拟机测试**：需要真实渲染、窗口或输入，但不需要代表性宿主硬件时使用。
3. **本机真机前台测试**：仅在确有必要且获得用户本次明确授权后使用。

“本机后台”是指不抢焦点、不注入宿主键鼠、不弹出前台窗口、不干扰用户工作的 headless、命令行、服务端、构建、协议和数据测试。

执行者与环境是两条独立维度：默认由 agent 完成测试；真人可以在 VM 或宿主机测试，但只有第 7.4 节规定的 Human Gate 才请求真人。会显示窗口的浏览器自动化不属于后台测试，必须进入 VM，或取得宿主真机租约。

### 7.2 本机真机租约

agent 只有在用户明确说明当前不使用机器、已经休息/睡觉，并对本次测试给出明确授权后，才能进行会抢占焦点、注入真实键鼠、打开前台 Godot/浏览器或影响日常使用的宿主真机测试。

授权规则：

- 单次授权，不从过去对话、用户暂时无操作或机器空闲推断。
- 测试前说明：为什么后台/VM 不够、候选版本、测试路径、预计时长、焦点影响、证据和清理方式。
- 测试期间只有一个 agent 拥有宿主输入与前台窗口。
- 输入或焦点不确定时立即停止，释放全部按键，只关闭本次测试创建的进程。
- 测试完成即清理并结束租约，后续真机测试重新申请。

没有授权时，agent 应继续所有不依赖真机的工作；只有真机结论成为当前阶段不可绕过的门禁时，才停下说明并等待。

### 7.3 虚拟机

- 虚拟机是前台真机之前的第二选择，用于隔离窗口、输入和截图。
- 不假定 `PayYourLife-TestVM` 自动属于本项目；必须使用项目明确分配的 VM 或先建立隔离规则。
- VM 能证明场景、输入和渲染路径，但不能自动代表宿主 GPU 性能、真实音频设备、控制器手感或网络环境。
- VM 测试结束后释放按键，关闭测试创建的窗口和进程，保存必要证据。

### 7.4 必须真人介入的测试

默认由 agent 完成所有能完成的测试，不得因为麻烦、耗时或自动化困难把测试推给用户。

通常需要真人的维度包括：

- 主观音色、混音舒适度、疲劳感和情绪是否合适
- 真实键鼠、手柄、触屏的长期操作手感与身体反馈
- 代表性宿主 GPU/显示器/音频设备上的真实性能和硬件问题
- 真实多人 PvP 的欺骗、博弈、紧张和社交压力
- 美术方向、玩法走向、商业定位等需要产品所有者拍板的选择
- 需要长时间真实游玩才能判断的耐玩性、疲劳与重复感

Goal Card 必须提前标出已知 Human Gate。到达该门禁时，agent 应提供：项目/候选、测试步骤、预计时长、期望结果、失败时可观察现象、已由 agent 验证的范围，以及为什么 agent/VM 不能完成。

只有真正必须真人介入时才请求；若该结论阻断阶段接受，则停止依赖工作并等待测试。仍有不依赖工作时先继续，不用真人等待 agent。

## 8. 项目测试分层

按被测对象选择最小真实环境。被测行为依赖几个系统，就开几个系统。

### 8.1 纯逻辑与合同

使用 Node/TypeScript 直接导入真实服务端或 shared 模块，构造最小 room/state：

- 战斗、掉落、背包、撤离、结算、档案和协议
- 权威性、重复结算、越权输入和版本迁移
- 不加载地图、浏览器或 Godot 场景

禁止复制一份游戏逻辑到测试脚本。测试必须调用真实产品代码。

### 8.2 单一机制沙盒

Phaser 旧客户端使用 `?devRoomPreset=sandbox` 及必要 test hooks，测试真实客户端和真实服务端，但控制环境：

- 移动、攻击、开箱、遮挡、单个资产、VFX、UI 和音频触发
- 无关怪物、全图压力和长时间流程应关闭

客户端自动移动使用 `window.__P0B_TEST_HOOKS__`，不要用无法进入 Phaser 输入层的合成 KeyboardEvent 冒充真实路径。

### 8.3 跨系统链路

结算、档案回流、死亡掉落、多人、撤离全程和黑市使用真实全局链路。只在阶段 checkpoint 或发布候选时运行对应全局 gate。

### 8.4 可见与可听改动

任何玩家看得见或听得见的改动，提交前必须检查实际渲染或声音证据：

- Phaser 可运行 `node scripts/accept-game-feel-baseline.mjs` 或规定的 agent-player 场景，并亲眼读取产出截图/连续帧。
- 查看 `.devlog/latest.jsonl`，确认预期事件真实发生且频率合理。
- 音频自动检查覆盖时长、峰值、RMS、静音、削波、重复触发与并发；主观听感仍属于 Human Gate。
- 契约绿、canvas 非空或单张静态截图不能替代动作和节奏验收。

### 8.5 Godot 测试

- 后台：命令行导入、脚本解析、资源、数据、存档、协议和 headless 场景烟测。
- VM：真实渲染、场景流、窗口输入、碰撞、遮挡、动画和截图/录像。
- 宿主真机：只有获得单次授权后，验证代表性 GPU、真实音频、物理输入和最终手感。

## 9. 真正的 2.5D 合同与 Godot 边界

未来客户端必须满足以下可观察条件，才可称为 2.5D：

- 固定斜视/正交相机，目标俯角约 50-60°，全场景投影一致。
- 世界存在明确地面平面与高度关系，移动、碰撞和导航在统一空间中工作。
- 建筑、桥梁、障碍、角色和怪物具有正确前后遮挡，不能只靠平面贴图和散乱 `setDepth(y)` 冒充。
- 人物、怪物、地表、建筑、箱子和 VFX 使用统一相机角度、比例、脚底锚点和左上主光。
- Sprite 角色至少为 idle / walk / attack / hurt / death 提供前、后、左、右四向可读表现；左右仅在武器、光向和剪影翻转后仍正确时允许镜像复用。使用真实 3D 模型或连续朝向动画时不强制四套贴图，但仍须验证四个基本方向。单侧图水平翻转不能替代前后方向。
- 角色不会因生成帧脚点漂移、程序 bob、网络插值或贴图切换产生悬浮和乱飘。

Godot 的作用是验证并承接更可靠的场景、动画、遮挡、音频和原生游戏 UI 工作流，不是为了更换技术栈而更换技术栈。

迁移原则：

- 保留 Node.js 权威服务端、现有搜打撤规则、档案经济和有价值的领域合同。
- 在仓库独立 `godot/` 目录和明确实验分支中建立候选，不逐文件翻译 TypeScript。
- Phaser 版本作为可运行 A/B Baseline；Godot 未通过 Go Gate 前，不删除或重写旧客户端。
- 不长期双线同步新内容。Godot 垂直切片通过后再决定唯一客户端生产主线。
- 首个切片必须是完整玩家节拍，不是空地图、技术展厅或功能等价复刻。
- Godot Go Gate 使用本文件第 6 节同一组四层门禁和 Goal Card 的 Canonical Path。独立 QA 必须通过 Functional；两名互相隔离的 Reviewer 必须各自达到 `75/100`、核心维度不低于 `3/5`，且都在匿名 A/B 中确认 Godot 有明确 Delta；Producer 才能作出 `ACCEPT` 并扩大迁移。否则为 `RETURN`、`PARTIAL`、`REJECT` 或 `NEEDS_REVIEW`。

## 10. 资产生产与集成

生图模型是资产生产能力，不是场景导演。高质量单图只有被正确集成，才算游戏资产。

每项商用候选资产必须记录：

- owner、用途和玩家出现时刻
- source path 与 runtime path
- 相机、光向、显示尺寸和风格参考
- 生成工具、prompt、negative prompt 和修改记录
- provenance、license status 和商业审查
- 状态：concept / placeholder / candidate / licensed / final / rejected
- introduced segment、expiry gate 和 replacement brief

资产文件与实际引用、碰撞/锚点、场景配置和必要验证应作为同一可验证增量提交。

禁止：

- 生成后只 preload、不在真实场景使用
- 用高精细立绘缩小成噪点
- 不同资产各自使用不同俯角、光向和比例
- 用程序圆圈、方块和调试标签替代正式世界反馈
- 用一张背景图承担本应由场景、遮挡、碰撞和导航承担的空间职责
- 因为 imagegen 可用就到处添加装饰，破坏信息层级

资产使用以“恰到好处”为准：首先服务世界可读性、路线、交互、动作和情绪，再服务装饰。

## 11. 动画、VFX 与音频阶段

动画、VFX 和音频可以安排集中阶段，但每个阶段仍需围绕具体玩家节拍，不做无边界“全面优化”。

动画要求：

- 明确脚底/根节点锚点和每帧重心。
- AnimationTree/状态机或等价单一所有者控制动作切换。
- 网络插值、角色朝向、局部受击挫动和程序动效不能争夺同一变换。
- 运动质量、前摇后摇和剪影可读性优先于单帧精细度。

VFX 要求：

- 每个效果有唯一事件所有者、明确层级和视觉预算。
- 反馈不能互相遮挡角色、战利品和危险信息。
- 不以程序星芒、圆环和叠字数量代替成就感。

音频要求：

- 建立 Master / Music / Ambience / SFX / UI 等明确总线。
- 每个事件只有一个音频所有者。
- 同类 cue 有冷却、并发上限、优先级和必要随机变化。
- 关键奖励、受击或撤离可触发 ducking，但不能让所有声音争抢注意力。
- 事件日志必须检查触发频率；短时间重复几十次即使音频文件正常也属于失败。

## 12. 架构纪律

`docs/REFACTOR-GUIDE.md` 的产品铁则和板块边界继续有效，但架构不能重新成为开发主线。

- 服务端继续作为战斗、掉落、死亡、撤离、结算和档案真相源。
- 客户端只展示已确认事实和即时本地表现，不自行发明权威结果。
- 改一个功能优先只动一个板块；跨板块事实通过结构化事件或明确接口传播。
- 不把新功能继续塞进 `GameScene`、总入口或万能 Manager。
- 同一反馈只能有一个运行时所有者，旧实现与新板块不能并行重复响应同一事件。
- 数值进入对应板块的数据/常量，不散落在渲染和网络回调。
- 不为满足文档中的 23 个板块而提前实现预留系统。
- 不做玩家无感知的大重构，除非它是当前 Player Promise 的明确阻塞，且范围被 Goal Card 限定。

画面、动作和核心手感未通过当前 Experience Gate 前，冻结无关新系统和内容扩张。

## 13. Git 与工作区

工作区可能包含用户、其它 agent 或历史测试产物。不得覆盖、回滚、删除或提交不属于当前任务的内容。

- 开始前检查 branch、status、recent log 和相关 checkpoint。
- 使用路径明确的 `git add <task paths>`，禁止默认 `git add -A`。
- 一个 commit 只承载一个清楚、可验证的玩家增量或必要工程底线。
- commit message 写明：玩家变化、原因、验证、未验证维度和 Human Gate。
- 用户授权写入的开发任务中，先确认 remote、目标 development branch 和推送范围；完成清楚且验证过的增量后及时 commit 并 push。只读任务不 commit/push，内部失败尝试和纯证据噪音不成为 master 提交。
- WIP/handoff commit 是“只提交可验证增量”的明确例外，仅在未完成工作必须跨会话保留时使用；推送到已确认的 development/experiment branch，不把未完成候选直接推入受保护发布分支。
- annotated tag 只用于可交接的 playable checkpoint，不用于每次 readiness 微调。
- 资产和引用/集成代码同一提交。
- 不使用 `git reset --hard`、`git checkout -- .`、`git clean -fd`、force push 或覆盖远端历史，除非用户明确批准并已报告影响。

push 是远端保存标志，不是产品验收标志。

## 14. 运行、日志与资源卫生

项目默认端口：

- 客户端：`5288`
- 服务端：`5289`
- 漂移端口优先使用 `52XX`，不使用 `5173` 或 `3000`

常用入口：

- `npm run playtest:manual`
- `npm run dev`
- `http://localhost:5288/`
- `http://localhost:5289/health`

Runtime Debug Log：

- 当前会话：`.devlog/latest.jsonl`
- 历史会话：`.devlog/session-<timestamp>.jsonl`
- 诊断用户报告前先读对应时间线，不先凭描述猜根因。
- `logEvent` 只增加高信号事件，不把日志量当覆盖率。

资源纪律：

- dev server、浏览器 tab、MCP、VM 会话、前台窗口和 agent 都是短生命周期资源。
- 用完关闭本任务创建的资源，不保留进程只为证明工作发生过。
- `.codex-artifacts` 只保留当前 checkpoint 所需证据，避免无限堆积；清理历史产物前必须确认归属并取得需要的授权。
- 不擅自杀死无法确认归属的 Node、浏览器、Godot、代理或系统进程。

## 15. 停止与升级

普通 bug、局部测试失败、可逆审美选择、空 backlog 或一个微型计划完成，都不是停止开发的理由。

必须停止并请求用户的情况：

- force push、删除大目录、覆盖历史、迁移/删除数据等不可逆操作
- 玩法方向、美术方向、平台、商业模式或引擎主线等重大产品决策
- 购买、账号、凭据、授权、公开发布或法律风险
- 当前阶段到达明确 Human Gate，且没有可继续的非依赖工作
- 同一 P0/P1 已用两种根因方案仍无 workaround
- 两个完整候选周期都没有产生有效 Experience/Delta
- 本地环境无法修复，且后台、VM 和安全替代方案均已穷尽

请求人工测试时必须具体，不得只说“请帮忙玩一下”。

## 16. 交接格式

每次非平凡阶段性交接必须简洁说明；不适用或未发生的字段省略：

- Player Promise 与玩家可感知变化
- Functional / Experience / Delta / Engineering Floor 状态
- Baseline、候选 branch、HEAD、提交和 checkpoint
- 当前 dirty state，以及哪些修改不属于本任务
- 已运行的后台、VM 或授权真机测试及其证据
- 尚未验证的 Human Gate 和准确原因
- 最多三个决定性剩余问题
- 精确恢复锚点和下一玩家可感知目标
- 当前阶段是 `ACCEPT`、`RETURN`、`PARTIAL`、`REJECT` 或 `NEEDS_REVIEW`

禁止用测试通过数量作为交接摘要的主体。
