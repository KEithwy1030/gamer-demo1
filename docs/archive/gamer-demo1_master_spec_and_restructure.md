## 0. 文档目的

这不是单纯的“开发计划”，而是 **Demo 1 的唯一执行总纲**。

它解决四个问题：

1. 把你们现在反复 Debug 的根因讲透。
2. 判断当前语言和框架是否走错方向，是否需要立即重构。
3. 把 Demo 1 的玩法、机制、前后端边界、执行顺序写成可落地规格。
4. 让本地 Agent 在没有额外上下文的情况下，也能按统一标准逐步推进项目。

本文件未来建议落地到仓库：

- `docs/DEMO1_MASTER_SPEC.md`

## 1. 先给结论：现在要不要换语言、换框架、重做仓库？

## 1.1 结论

**不建议现在为了 Demo 1 立即重写到另一套语言或引擎。**

当前方向：
- `client`: TypeScript + Phaser 3 + Vite
- `server`: Node.js + Socket.IO
- `shared`: 前后端共享类型/协议/静态数据

这条路线对于你现在这个项目目标是成立的：
- 游戏是 **2D 俯视角 / 2.5D 视觉伪装**；
- 首发目标是 **Web 可玩、局域网/房间制联机 Demo**；
- 功能核心是 **搜 → 打 → 撤** 的短局循环；
- 项目最痛的不是图形能力不足，而是 **规格漂移、共享契约漂移、测试闭环不足**。

所以现在真正错误的，不是语言，而是开发治理方式。

## 1.2 为什么不建议立刻重写

因为你们目前的主要问题并不是：
- Phaser 做不到；
- TypeScript 做不到；
- Socket.IO 做不到；
- Node.js 性能已经成为 Demo 1 的瓶颈。

真正的问题是：

1. **设计没有唯一真源**。  
2. **client/server/shared 没有形成严格冻结的共享契约**。  
3. **文档、代码、测试脚本三者不总是同一版本的世界观**。  
4. **可见反馈不足，导致测试反馈模糊，Agent 误判状态。**  
5. **执行顺序没有被强约束，导致一边改功能、一边修 bug、一边补文档。**

如果这几个问题不先解决，就算换到 Godot、Defold，甚至任何更“像游戏开发工具”的引擎，最后也仍然会重复同样的问题。

## 1.3 什么时候才应该考虑重构到另一套技术路线

只有出现下面任一情况，才建议在 Demo 1 之后评估引擎迁移：

1. 项目不再是 Web-first，而是明确变成 **移动原生优先**。  
2. 地图、UI、动画、碰撞和关卡内容开始高度依赖 **可视化编辑器工作流**。  
3. 你们希望更强的 **场景编辑、碰撞可视化、UI 编辑器、动画状态机** 来替代大量手写前端逻辑。  
4. 后续版本要走到更复杂的移动性能优化、资源分包、编辑器驱动内容生产。  
5. 团队不再想维护“浏览器游戏 + Web 包装移动端”的路径，而想直接上原生工作流。

在 Demo 1 阶段，上面这些条件都还不构成立刻重写的收益。

## 2. 当前仓库的真实工程判断

## 2.1 当前仓库不是“废架构”，而是“半成品但方向对”

当前仓库已经形成了正确的三层结构：

- `shared/`：协议、类型、静态数据、共享常量
- `server/`：房间、战斗、怪物、掉落、撤离、结算、库存等权威逻辑
- `client/`：Lobby、GameScene、实体表现、HUD、背包、结果页

这说明基础方向是对的。

## 2.2 当前最核心的工程问题

### 问题 A：多份真相并存

现在至少存在以下几层“事实源”：
- GDD
- README
- `docs/agent/PROJECT_STATE.md`
- `docs/agent/OPEN_LOOPS.md`
- `docs/agent/DECISIONS.md`
- `shared/` 当前常量和类型
- `server/` 当前真实逻辑
- `client/` 当前真实入口和表现

这些内容没有被严格排序，导致新 Agent 很容易基于错误前提继续开发。

### 问题 B：shared 层消费方式不够统一

根据现有仓库分析，当前存在两类风险：
- 客户端直接吃 `shared/src/...`
- 服务端在某些路径上依赖 `shared/dist/...`

这会造成“客户端看到最新源码、服务端可能跑旧产物”的幽灵 Bug。

### 问题 C：UI 和服务端规则不总是同构

典型表现：
- 服务端库存/掉落模型已经支持尺寸和占格，但前端背包 UI 仍可能按线性格子思路渲染；
- 服务端技能能力比客户端暴露的按钮/按键更完整；
- 服务端撤离/结算链路已经较完整，但前端“返回大厅”等收口动作仍容易残留状态。

### 问题 D：测试覆盖偏协议，偏后端

现在已有自动化主链路脚本，这很好；但它更偏：
- 协议层是否能跑通；
- Socket.IO 主链路是否正确；
- 服务端状态流是否闭环。

它不能替代：
- 浏览器端 UI 冒烟；
- 反馈是否清楚；
- 结果页和回大厅是否干净；
- 移动端输入是否可用。

### 问题 E：内容设计范围容易失焦

项目真正想做的是 Demo 1，但文档中同时存在一些长线野心：
- 交易系统
- 商业化后端
- 更复杂的技能体系
- 更大规模匹配

这些不应该进入 Demo 1 的交付面。

## 3. Demo 1 的唯一产品定义

这一节是 **Demo 1 的产品真源**。

## 3.1 Demo 1 的一句话定义

Demo 1 是一个 **Web-first、房间制、多人联机、2D 俯视角“搜打撤”垂直切片**。

它只验证一件事：

> 玩家能否在同一局地图中完成“搜资源 → 打怪/对抗 → 背包取舍 → 前往撤离点 → 撤离或失败结算”的完整闭环。

## 3.2 Demo 1 只做什么

必须交付：
- 创建房间 / 加入房间
- 多玩家进入同一局
- 玩家移动
- 普攻
- 三种武器差异
- 怪物（普通 / 精英）
- 掉落
- 背包格子系统
- 装备与卸下
- 消耗品使用
- 玩家死亡全掉落
- 固定撤离点
- 成功/失败结算
- 返回大厅并可继续下一局

## 3.3 Demo 1 明确不做什么

以下内容一律不进入 Demo 1：
- 交易行
- 保险箱
- 打造
- 强化
- 复杂账号系统
- 复杂匹配系统
- 聊天系统
- 世界 Boss
- 撤离道具
- 自由技能搭配
- 多职业系统
- 商业化后端

这些都是 Demo 2 以后再讨论的事。

## 4. Demo 1 的玩法与规则规格

> 本节是产品规则。只要代码和这里冲突，以本节为准；若代码暂未做到，则进入开放项，不允许私自变更产品定义。

## 4.1 基础类型

- 类型：2D 俯视角搜打撤原型
- 平台：Web 浏览器优先
- 视觉目标：2D 实现 2.5D 感受，不追求真 3D
- 模式：PVP + PVE
- 联机方式：房间制
- 原型阶段人数：2~4 人可验证，但架构保留多人房间制

## 4.2 单局目标

玩家进入同一张地图后：
- 搜物资
- 打怪
- 发生少量 PVP
- 拾取、装备、管理背包
- 在撤离开放后前往固定撤离点
- 成功带出或失败结算

## 4.3 单局节奏

建议保留 GDD 的三段节奏：

### 阶段一：0 ~ 5 分钟
- 玩家分散出生
- 搜刮基础资源
- 打普通怪
- 少量遭遇战

### 阶段二：5 ~ 10 分钟
- 玩家逐渐向中高价值区汇聚
- 精英怪、优质掉落成为争夺核心
- PVP 增加

### 阶段三：10 ~ 15 分钟
- 固定撤离点开放
- 幸存玩家围绕撤离点争夺
- 未撤离则结算失败

## 4.4 成功与失败条件

成功：
- 在时间结束前进入固定撤离区
- 完成撤离读条
- 未被打断
- 带出当前背包与穿戴物品

失败：
- 被怪击杀
- 被玩家击杀
- 到时间前未撤离

## 4.5 地图与空间规格

当前基线建议先按 GDD 锁定：
- 地图：`4800 x 4800 px`
- Tile：`32 x 32 px`
- 视口：PC 默认 `1280 x 720`
- 固定撤离点：地图中心
- 出生点最小间距：`800 px`

> 注意：如果当前代码实现与此不一致，不允许口头修改，必须在阶段 P0 中由集成人把真实值对齐进本文件和 `shared/`。

## 5. 战斗系统规格

## 5.1 玩家基础属性

以 GDD 为当前基线：
- 生命：100
- 基础移动速度：5 px/帧（60fps 基准）
- 暴击率：0%
- 暴击伤害：150%
- 闪避率：0%
- 基础减伤：0%

## 5.2 闪避

闪避技能与闪避率属性必须分开：

### 闪避技能
- 位移：150 px
- 无敌帧：0.2 秒
- 冷却：5 秒

### 闪避率属性
- 来自装备词条
- 表示被攻击时概率性规避伤害
- 不得与主动闪避技能混淆

## 5.3 武器定位

按当前 GDD 的 Demo 1 产品意图：

### 剑
- 高频、灵活、贴身压制
- 低伤高频

### 刀
- 均衡、正面对拼
- 中等距离与中等节奏

### 枪
- 高爆发、低频、惩罚失误
- 更长攻击距离

## 5.4 武器数值处理原则

这里必须强调：

**GDD 数值不是直接等于当前代码真值。**

处理规则：
- 产品层先保留 GDD 的相对定位；
- 工程层在 P0 统一真实运行参数；
- 所有 README / PROJECT_STATE / OPEN_LOOPS / `shared/src/data/weapons.ts` 都必须对齐为唯一值；
- 测试脚本禁止再硬编码旧攻击距离。

## 5.5 技能策略

当前不要一口气把每把武器全部技能做满。

推荐：
- 主武器先完整做 1 套稳定技能 + 普攻 + 闪避
- 另两把武器先做到“可测、可用、不坏链路”
- 等 Demo 1 主链路稳定后，再补技能差异化表现

换句话说：

> Demo 1 的敌人不是“技能不够多”，而是“每把武器都只完成一半”。

## 6. 怪物、掉落、背包与装备规格

## 6.1 怪物

保留两类：
- 普通怪
- 精英怪

AI 只做简单可用逻辑：
- 巡逻
- 玩家进入仇恨范围即追击
- 不做呼叫同伴
- 失去仇恨后停止或回待机

不要在 Demo 1 上复杂寻路。

## 6.2 掉落

必须存在：
- 金币
- 武器
- 防具
- 珍品
- 消耗品（若当前已接入）

掉落链路要求：
- 服务端生成
- 客户端只展示
- 拾取必须服务端校验

## 6.3 玩家死亡

玩家死亡后：
- 背包内容全部掉落
- 穿戴武器、防具全部掉落
- 不存在保险槽或保护栏

## 6.4 背包

当前建议规格：
- `10 x 6 = 60 格`
- 前端必须按占格真实渲染
- 不允许再把背包抽象成仅用于演示的线性列表

必须支持：
- 放不下时不可拾取
- 明确提示“背包已满”
- 装备 / 卸下 / 丢弃 / 使用

## 6.5 装备系统

Demo 1 保留五个槽位：
- 武器
- 头
- 胸
- 手
- 鞋

这已经足够支持搜刮和取舍，不要扩更多部位。

## 7. 撤离与结算规格

## 7.1 撤离

当前建议固定为：
- 仅一个固定撤离点
- 第 10 分钟开放
- 圆形撤离区
- 读条 5 秒
- 受击 / 离开区域 / 死亡任一条件会中断

## 7.2 撤离触发方式

这里必须冻结，不允许“文档一种、代码一种、UI 又是一种”。

二选一，只能选一个：

### 方案 A：自动开始
- 玩家进入撤离区后自动开始读条
- 优点：实现简单，测试成本低
- 缺点：搜打撤味道略弱

### 方案 B：手动触发
- 玩家进入撤离区后，需要主动按键/按钮开始读条
- 优点：更像搜打撤
- 缺点：需要额外输入和提示设计

**建议：Demo 1 优先使用自动开始。**

原因：
- 先把链路收紧；
- 后续再决定是否升级到更强交互语义。

## 7.3 结算

成功结算必须展示：
- 存活时长
- 玩家击杀数
- 怪物击杀数
- 带出物品
- 金币
- 珍品价值
- 返回大厅按钮

失败结算必须展示：
- 失败原因
- 存活时长
- 击杀数
- 物品掉落/丢失说明
- 返回大厅按钮

## 7.4 返回大厅

这不是边缘功能，而是 Demo 1 交付定义的一部分。

必须保证：
- Socket 正确离房
- 场景正确销毁
- DOM overlay 清理干净
- 上一局背包/结果页状态不残留
- 不刷新页面也能开下一局

## 8. 前后端与 shared 的职责边界

## 8.1 shared

只负责：
- 事件名
- payload 类型
- 共享枚举
- 武器/物品/怪物等静态配置
- 地图元数据（如障碍定义、出生点定义）

禁止：
- 场景逻辑
- 服务端运行时状态
- 带副作用逻辑

## 8.2 server

只负责：
- 房间状态
- 玩家最终位置合法性
- 战斗结算
- 怪物状态
- 掉落生成与拾取校验
- 撤离成功判定
- 结算结果

凡是影响比赛结果的事情，都必须由 server 决定。

## 8.3 client

只负责：
- 输入采集
- 场景渲染
- HUD
- 实体表现
- 插值和平滑
- 反馈与动画
- 结果页展示

客户端可以预测，但不能定义真实结果。

## 9. 以后所有 Agent 必须遵守的治理规则

## 9.1 Agent 进入仓库的阅读顺序

任何 Agent 开始改代码前，必须先按顺序读取：

1. `docs/DEMO1_MASTER_SPEC.md`（本文件）
2. `AGENTS.md`
3. `docs/agent/STATUS.json`
4. `docs/agent/PROJECT_STATE.md`
5. `docs/agent/OPEN_LOOPS.md`
6. `docs/agent/DECISIONS.md`
7. `EXECUTION_PLAN.md`

## 9.2 真相优先级

发生冲突时，优先级固定为：

1. 本文件 + 当前 `shared/` 真值
2. `docs/agent/DECISIONS.md`
3. `docs/agent/PROJECT_STATE.md`
4. 运行代码
5. README
6. 历史 handoff

## 9.3 每次任务必须先写变更声明

模板：

```md
任务：
会改文件：
不会改文件：
是否修改 shared：是 / 否
验收方式：
回归风险：
```

## 9.4 一次只允许做一类变更

每次任务只能属于以下一种：
- 表现层任务
- 权威逻辑任务
- 协议同步任务
- 文档/测试任务
- 数值调整任务

禁止一次任务同时修改：
- shared 契约
- server 权威逻辑
- client 表现入口

除非这次任务被明确标记为“协议同步任务”。

## 9.5 共享层只能由一个 Agent 改

如果多 Agent 并行：
- Agent A：只负责 `shared/` + 主规格文档
- Agent B：只负责 `server/`
- Agent C：只负责 `client/`
- Integrator：只负责集成、验证、更新 `docs/agent`

## Phase P1：shared 契约统一与消费方式清理

### 目标

让 client/server/shared 在开发态和构建态都读取同一套共享定义。

### 必做

1. 检查所有 `shared` 消费入口。

## 13. 语言与工具的长期建议

## 13.1 对当前仓库的建议

**保持当前 TypeScript Monorepo 路线，不重写。**

理由：
- 现在最关键的是统一规则和边界，不是换武器；
- Web-first 2D 联机 Demo 与 Phaser/Socket.IO 的匹配度足够；
- shared 层是你们当前降低联机契约错误的最大资产；
- 立即换引擎会让 networking、背包、场景、输入、结果页全部重新开一遍。

## 13.2 当前最适合你的免费开发组合

### 方案 A：继续当前路线（推荐）
- 游戏：Phaser 3
- 构建：Vite
- 联机：Socket.IO
- 地图：Tiled
- UI 回归：Playwright
- 移动端壳：Capacitor（仅在需要安装包时引入）

这是对你现在最省成本的方案。

### 方案 B：下一个项目再考虑 Godot

如果你下一阶段明确要：
- 更重可视化编辑器；
- 更强移动原生工作流；
- 更少手写 HUD / 场景 / 碰撞可视化；

那可以在 Demo 1 做完后评估 Godot。

### 方案 C：极轻量快速原型可考虑 GDevelop

但只适合：
- 更偏玩法验证；
- 更少自定义网络协议；
- 更少自定义服务端权威逻辑。

对你当前这个项目，不是最优主线。

### 方案 D：Defold 只在接受 Lua 时考虑

如果团队接受 Lua，且想要：
- 小体积
- 一体化编辑器
- 免费、无版税、跨平台

Defold 也值得作为后续评估对象。

## 13.3 一个很重要的现实判断

你现在感觉“只靠 AI CLI 开发有局限”，这个判断是对的。

原因不是 AI 不够强，而是 CLI 天生不擅长这些东西：
- 观察 UI 手感
- 判断空间感
- 判断碰撞是否违和
- 判断按钮可见性和触控热区
- 判断结果页是否真的清楚

所以正确做法不是抛弃 AI CLI，而是给它补上两个外脑：

1. **主规格文档**：防止需求漂移  
2. **浏览器自动化 + 人工冒烟**：防止“代码没报错但体验已坏”
