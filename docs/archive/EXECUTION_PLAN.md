# Demo 1 执行文档 v2

基于设计文档：[GDD_Demo1_v1.3.docx](/E:/CursorData/gamer/GDD_Demo1_v1.3.docx)

## 1. 文档修订原因

上一轮实现把“联机骨架跑通”误当成了“可试玩 Demo”。这是错误的验收口径。

当前修订后的硬门槛是：

- Demo 必须先成立为一个可辨识的 2D 游戏画面。
- 玩家一进入场景，必须直观理解“这是游戏场景，不是调试界面”。
- 联机、战斗、掉落、背包、撤离这些系统，只有在 2D 游戏画面成立后才算有意义。

这份文档的目标不是继续铺系统，而是先把当前工程底座收成一个可供测试人员游玩的 2D 垂直切片。

## 2. 新的交付目标

本轮交付目标定义为：

`局域网联机 + 2D 游戏画面成立 + 一条完整主链路可试玩`

主链路固定为：

`创建/加入房间 -> 开局进入同一张 2D 地图 -> 移动/攻击 -> 打怪 -> 掉落 -> 拾取 -> 前往撤离区 -> 撤离成功/失败结算`

本轮不追求：

- 大而全的系统覆盖
- 精美美术
- 完整商业化 UI

本轮必须追求：

- 2D 地图和角色表现
- 清晰的空间关系
- 可理解的操作目标
- 测试员不看代码也能完成一局试玩

## 3. 新的最低验收标准

以下条件全部满足，才能称为“可试玩 Demo”。

### 3.1 画面成立

- 有明确的 2D 地面层，不是纯色空场。
- 有清晰可辨识的地图边界、障碍、通路、出生区、撤离区。
- 玩家、怪物、掉落物都必须是独立的 2D 实体表现，不允许只靠文本和抽象标记代替。
- 镜头跟随和空间比例正常，玩家能理解自己在地图中的位置。

### 3.2 游戏语义成立

- 玩家一眼能区分：
  - 自己
  - 其他玩家
  - 敌人
  - 掉落物
  - 撤离点
- 攻击、受击、死亡、拾取、撤离必须有基础视觉反馈。
- HUD 必须是游戏 UI，不允许用大段调试文本充当主要信息承载。

### 3.3 试玩链路成立

- 两台同一局域网设备可以进入同一房间。
- 房主开局后，双方进入同一张地图。
- 玩家可移动、攻击、打怪、拾取。
- 撤离区可见且可交互。
- 一局最终能进入成功或失败结算。

## 4. 本轮范围收缩

为了尽快形成真正可试玩的 2D 切片，本轮做减法。

### 4.1 必做

- 局域网联机房间
- 单张小地图
- 2D 场景地面和障碍
- 玩家 2D 表现
- 普通怪 1 种，精英怪 1 种
- 掉落物基础表现
- 撤离点基础表现
- 一把武器的完整战斗手感
- 简化背包显示与拾取
- 成功/失败结算

### 4.2 暂缓

- 三武器全部做完
- 九个专属技能全部做完
- 复杂背包拖拽
- 全套装备成长
- 大地图内容密度
- 高级特效和精细美术

## 5. 视觉与交互重构原则

### 5.1 2D 画面原则

- 优先使用简洁但明确的 2D 形状语言。
- 即使没有正式美术资源，也必须像一个俯视角 2D 游戏，不像调试器。
- 地图必须由多个视觉层组成：
  - 底地面
  - 路径/区域差异
  - 障碍或掩体
  - 交互点

### 5.2 实体表现原则

- 玩家：
  - 自己与他人颜色区分明显
  - 具备朝向感和脚下定位感
- 怪物：
  - 普通怪与精英怪轮廓和颜色不同
  - 不再只是一块抽象标记
- 掉落：
  - 必须明显可拾取
  - 名称显示简短，不遮挡主视野
- 撤离点：
  - 场景中显眼
  - 进入范围、可撤离、撤离中三种状态可识别

### 5.3 HUD 原则

- 主 HUD 只保留必要内容：
  - 房间号
  - 倒计时
  - 当前生命
  - 当前武器
  - 撤离状态
- 调试信息不再作为主要 UI。
- Inventory 面板在局内可保留，但不能抢占主视野，也不能像页面主内容。

## 6. 技术实现策略

### 6.1 保留不动的底座

- `server` 的房间、同步、怪物、掉落、背包、撤离基础逻辑继续保留。
- `shared` 协议和基础数据继续保留。
- `client` 的 Socket 接线、Lobby 流程、结果页基础流程继续保留。

### 6.2 本轮主要改造层

- `client/src/scenes/GameScene.ts`
  - 从调试场景改造成真正的 2D 游戏场景
- `client/src/game/entities/*`
  - 重构玩家、怪物、掉落表现
- `client/src/styles/*`
  - 收紧 HUD、Inventory、结果页的视觉优先级
- `client/src/scenes/createGameClient.ts`
  - 保证场景挂载、尺寸同步、开局显示稳定

### 6.3 美术资源策略

本轮允许以下实现方式：

- 纯 Phaser 图形绘制
- 简单占位 sprite
- 低成本 tile/block 风格

但不允许：

- 只有纯文字和几何调试标记
- 纯空背景加面板

## 7. 并发重构拆分

主 session 只负责：

- 修订文档
- 明确验收口径
- 分配子任务
- 集成与验收

### Worker A：场景与地图重构

负责文件范围：

- `client/src/scenes/GameScene.ts`
- 允许必要时改 `client/src/scenes/createGameClient.ts`

目标：

- 把当前空场改成真正的 2D 地图空间
- 增加地面层、障碍、撤离区、出生区的可见结构
- 收紧 HUD，使主视野更像游戏
- 保证进入场景后画布尺寸与镜头正常

### Worker B：实体与表现层重构

负责文件范围：

- `client/src/game/entities/PlayerMarker.ts`
- `client/src/game/entities/MonsterMarker.ts`
- `client/src/game/entities/DropMarker.ts`
- 允许必要时改 `client/src/styles/inventory.css`
- 允许必要时改 `client/src/ui/InventoryPanel.ts`

目标：

- 让玩家、怪物、掉落成为明确的 2D 游戏实体
- 增强命中、可拾取、身份区分的视觉表达
- 让 Inventory 保持可用，但不再像页面主体

## 8. 新的阶段验收

### 阶段 A：画面验收

- 开局后截图看起来必须像 2D 游戏场景。
- 不能出现“页面里只有 Inventory”的观感。
- 不允许玩家需要靠代码或说明才知道自己在看什么。

### 阶段 B：局域网联机验收

- 两台设备访问 `http://<LAN_IP>:5173/`
- 一台创建房间，一台加入房间
- 房主开局后双方能进入同一场景

### 阶段 C：主链路验收

- 能打怪
- 能看到掉落
- 能拾取
- 能找到撤离点
- 能完成一局结算

## 9. 当前执行顺序

1. 重写执行文档，修正目标与标准
2. 并发重构场景层与实体表现层
3. 主 session 集成并修正运行问题
4. 用局域网地址完成双页实测
5. 再决定是否继续扩展武器、技能和背包深度

## 10. 当前结论

当前代码库不是废掉重来，而是：

- 联机技术底座可复用
- 表现层必须重构
- 验收标准必须前置到“2D 游戏画面成立”

后续所有“可开始测试”的结论，都必须以本文件第 3 节的标准为准。
