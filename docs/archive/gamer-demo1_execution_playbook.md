# gamer-demo1 执行文档（面向本地 Agent）

更新日期：2026-04-20  
适用对象：本地多 Agent / CLI Agent / 人工集成者  
目标：把当前仓库收敛成一个 **可持续开发、低反复 Debug、可稳定推进 Demo 1** 的工程。

---

## 0. 这份文档解决什么问题

当前仓库不是“完全没做完”，而是已经有一条能跑通的链路，但存在明显的“多份真相并存”问题：

- 设计文档（GDD）有一套目标。
- README 有一套说明。
- `docs/agent/` 有一套“当前真相”。
- `shared/`、`client/`、`server/` 又各自存在局部漂移。

重复 Debug 的核心原因，不是单纯“模型写错代码”，而是：

1. **规格没有唯一真源**。
2. **前后端共享契约没有被严格冻结**。
3. **自动化验证覆盖到协议，但没有覆盖可见 UI 与交互语义**。
4. **视觉层、输入层、服务端权威层，经常不同步推进**。

这份文档的目的就是：

- 建立一套 **单一真源 + 固定执行顺序 + 固定验收门槛**。
- 让本地 Agent 以后不再“看一点代码就直接改一大片”。
- 让每一轮工作都能落在清晰边界内。

## 1. 当前工程的真实结构

### 1.1 Monorepo 结构

- `client/`：TypeScript + Phaser 3 + Vite，负责大厅、场景、实体表现、背包 UI、结果页。
- `server/`：Node.js + Socket.IO，负责房间、对局、战斗、怪物、掉落、撤离、结算。
- `shared/`：共享类型、协议、静态配置。

### 1.2 当前已经可复用的核心能力

1. **房间链路**
   - 创建房间 / 加入房间 / 开局进入同一局。

2. **服务端主循环**
   - 玩家状态广播
   - 怪物 tick
   - 掉落同步
   - 撤离进度与结算

3. **库存/掉落系统底座**
   - 拾取、装备、卸下、丢弃、死亡掉落、消耗品使用
   - 词条/稀有度/装备属性投影已经有基础实现

4. **怪物生命周期**
   - 生成、仇恨、死亡、尸体停留、延迟重生

5. **自动化脚本**
   - 已有后端 Socket.IO 主链路脚本，可覆盖 create -> join -> start -> combat -> pickup -> extract -> settlement

### 1.3 当前最危险的漂移点

#### A. Shared 契约消费方式不统一

- `client` 里有直接从 `shared/src/...` 读取的情况。
- `server` 里大量从 `../../shared/dist/...` 读取。

这会导致：

- 客户端看到的是“最新源码”。
- 服务端可能跑的是“上一次 build 的 shared/dist”。

这是**高概率制造幽灵 Bug**的根因之一。

#### B. 背包模型和背包 UI 不一致

服务端库存逻辑已经是“带尺寸和占格”的模型，但前端背包 UI 仍然按固定 100 格线性渲染，容易让测试和 Agent 误判“系统已对齐”。

#### C. 服务端能力和客户端操作入口不一致

当前服务端已经实现了更多武器技能，但客户端技能映射和按钮/按键暴露并没有完全同步。

#### D. 设计说明与当前运行参数漂移

包括但不限于：

- 地图尺寸
- 撤离开放时间
- 武器攻击距离
- 怪物密度与刷新方式
- 出生点逻辑

#### E. 可见反馈还没有封口

攻击、受击、拾取、撤离这些关键反馈还不够强，导致“协议通了，但试玩仍像调试界面”。

## 2. 以后所有 Agent 必须遵守的硬规则

## 2.1 单一真源规则

新增一个文件作为**唯一开发规格源**：

- `docs/DEMO1_SPEC.md`

它负责定义：

- 当前 Demo 1 的唯一玩法规格
- 当前运行参数
- 当前保留功能
- 当前明确不做的功能

以后优先级固定为：

1. `docs/DEMO1_SPEC.md`
2. `docs/agent/DECISIONS.md`
3. `docs/agent/PROJECT_STATE.md`
4. 代码
5. README
6. 历史 handoff / 口头描述

README 不再承担“唯一事实源”职责，只做对外说明。

## 2.2 契约冻结规则

只要涉及以下内容，必须先改 `shared/`，再改 client/server：

- Socket 事件名
- payload 字段名
- 共享枚举
- 武器/怪物/地图等全局常量
- 背包尺寸/格子规则

任何 Agent **不得先改 client 或 server，再补 shared**。

## 2.3 一次只允许一个 Agent 改“共享层”

角色分工：

- **Agent A（Spec/Shared Owner）**：只负责 `docs/DEMO1_SPEC.md` + `shared/`
- **Agent B（Server Gameplay）**：只负责 `server/`
- **Agent C（Client Scene/UI）**：只负责 `client/`
- **Integrator（唯一集成人）**：合并、回归、更新 docs/agent

如果没有人工集成人，就指定一个主 Agent 承担 Integrator 角色。

## 2.4 每次改动都必须带“变更声明”

每次任务开始前，Agent 必须先写：

- 目标文件
- 不会碰的文件
- 是否改 shared 契约
- 验收方式

模板：

```md
任务：修正背包 UI 为真实网格渲染
会改：client/src/ui/**, client/src/styles/**
不会改：server/**, shared/**
共享契约是否变化：否
验收：build + 本地手动拖入/拾取/装备/丢弃检查
```

## 2.5 每次完成后都必须更新 repo memory

必须同步更新：

- `docs/agent/PROJECT_STATE.md`
- `docs/agent/OPEN_LOOPS.md`
- `docs/agent/WORKLOG.md`

否则后续 Agent 会继续基于过期上下文操作。

## 阶段 0：收口规格与共享契约（P0）

### 目标

把“多份真相”先压成一份真相。

### 必做项

1. 创建 `docs/DEMO1_SPEC.md`
2. 明确以下内容的唯一值：

   - 单局时长
   - 撤离开放时间
   - 撤离半径 / 读条时间
   - 玩家出生规则
   - 怪物数量 / 刷新规则
   - 三武器攻击距离 / 攻击频率 / 当前保留技能
   - 背包尺寸与占格规则
3. 检查并统一共享消费方式：
   - 禁止 server 继续直接依赖 `../../shared/dist/...` 作为开发态真源

4. 把 `scripts/test-loop.mjs` 的硬编码参数移出：

- 本阶段不要做视觉 polish
- 不要加新玩法
- 不要大改地图美术

### 验收标准

- `docs/DEMO1_SPEC.md` 落地
- client/server/shared 三端的关键常量一致
- `npm run typecheck`
- `npm run build`
- `scripts/test-loop.mjs` 不再依赖过期硬编码核心数值

## 阶段 1：背包/装备/掉落闭环对齐（P0）

### 目标

让“拾取 -> 背包显示 -> 装备/卸下 -> 丢弃 -> 使用消耗品”成为**视觉和数据都一致**的一条链。

### 必做项

1. 前端背包 UI 改成真实格子布局
   - 按 `x/y/width/height` 渲染
   - 不再使用固定 100 格线性列表
2. UI 标题、容量说明、格子占用规则全部与规格一致
3. 补“背包满”“无法拾取”“装备成功/失败”可见反馈
4. 消耗品使用后，数值变化与 UI 状态同步

### 推荐文件范围

- `client/src/ui/InventoryPanel.ts`
- `client/src/styles/inventory.css`
- `client/src/game/matchRuntime.ts`
- 如有必要，少量改 `shared/src/types/inventory.ts`

### 验收标准

- 真实显示多格物品占位
- 背包满时不可拾取且提示明确
- 装备/卸下/丢弃/使用全部可见且状态一致
- 不破坏后端自动化链路

## 阶段 2：输入层与技能层对齐（P0）

### 目标

把客户端可操作入口和服务端真实能力对齐。

### 必做项

1. 明确 Demo 1 当前保留的技能集
   - 如果服务端实现不全，就按 Demo 1 可玩优先，只保留每武器 1~2 个最稳定技能
   - 不要为了“看起来完整”强上全部技能
2. 客户端技能映射统一
   - 键盘
   - 移动端按钮
   - HUD 提示
3. 明确撤离交互语义
   - 二选一：
     - **手动触发**（更符合搜打撤）
     - **进入圈内自动开始**（更省实现）
   - 规格、README、UI 文案、代码必须一致
4. 闪避技能是否进入 Demo 1
   - 如果进：必须有明确无敌帧/位移/冷却表现
   - 如果不进：从规格和 UI 中临时移除

### 验收标准

- 三武器至少都能稳定触发一套明确技能入口
- 移动端与桌面端输入语义一致
- 撤离开始方式不再“文档一种、代码另一种”

## 阶段 3：战斗反馈、拾取反馈、撤离反馈封口（P0）

### 目标

让试玩者不看代码也能理解发生了什么。

### 必做项

1. 补全客户端反馈空实现

   - hit stop
   - pickup pulse / floating text

2. 玩家/怪物受击反馈统一
3. 撤离读条可见化

   - 中断原因
   - 成功提示
4. HUD 去调试味
   - 保留必要信息
   - 删掉让玩家困惑的大段状态文案

### 验收标准

- 攻击命中与未命中可区分
- 捡到物品时玩家能明确感知
- 撤离开始/中断/成功一眼可懂

## 阶段 4：出生点、怪物分布、障碍碰撞对齐（P1）

### 目标

解决“地图可读，但逻辑不可信”的问题。

### 必做项

1. 统一玩家出生策略
   - 现在实际出生逻辑必须与规格一致
   - 如果采用“中心外圈散开”，就把文档改成这一版
   - 如果采用“边缘分散出生”，就把 room-store 改过去
2. 统一怪物排除区规则
   - 排除区要对应真实玩家出生区域，而不是旧坐标假设
3. 障碍碰撞做成权威规则
   - 推荐：**服务端权威 + 客户端本地预览**
   - 先做 server 校验，再做 client 手感优化
4. 障碍布局数据化
   - 不要把大量障碍直接写死在场景逻辑里
   - 建议抽到 `shared/src/data/map/` 或 JSON 配置

### 验收标准

- 玩家不能穿过应阻挡的障碍
- 怪物分布与出生区关系合理
- 两台设备联机时位置不会因碰撞逻辑而明显分叉

## 7. 关于“要不要换工具/换开发方式”

## 7.1 对当前仓库的判断

**不建议为了 Demo 1 中途换引擎。**

原因：

- 你当前不是“什么都没有”，而是已经有房间、状态同步、战斗/掉落/撤离主循环底座。
- 真正拖慢进度的不是 Phaser 或 Web，而是规格漂移、共享层漂移、可见反馈没封口。
- 中途换 Godot/Defold/GDevelop，会把“调架构漂移”变成“重做所有运行时与联机集成”。

结论：

- **这个仓库继续用 Phaser + Socket.IO 做完 Demo 1 是最便宜的路径。**
- 先做完 Demo 1，再决定 Demo 2/正式版是否迁移。

## 7.2 如果从零做一个类似 Demo，免费路线怎么选

### 路线 A：Phaser + Vite + Socket.IO

适合：

- Web 优先
- 团队更熟 TS / 前端
- 需要快速联机原型

优点：

- 上手快
- Web 发布最直接
- 与现有前端工具链兼容最好
- 最适合现在这个项目的延续

缺点：

- 没有强可视化编辑器
- 地图、碰撞、动画状态、资源组织容易散
- 只用 CLI 开发时，非常依赖人脑保持结构纪律

### 路线 B：Godot

适合：

- 想要更强的可视化场景编辑
- 想减少“CLI 纯文本开发”的认知负担
- 后续可能兼顾原生移动端

优点：

- 场景、UI、碰撞、动画调试更直观
- 对“简单 2D demo + 可见调试”非常友好

缺点：

- 当前仓库迁移成本高
- Web 端导出与浏览器限制要额外注意

### 路线 C：Defold

适合：

- 想要轻量级、完整编辑器、跨平台
- 可接受 Lua

优点：

- 工具链完整、体量轻、导出范围广

缺点：

- 团队若主要写 TS，会有语言切换成本

### 路线 D：GDevelop

适合：

- 非常轻量的玩法验证
- 想更偏视觉搭建而不是代码

不适合：

- 复杂自定义联机逻辑主导的项目

## 9. AI CLI 的正确定位

不要把 AI CLI 当成“游戏开发环境本身”。

它更适合承担：

1. **代码生成器**
2. **规格对齐检查器**
3. **测试脚本编写器**
4. **重构助手**
5. **日志/差异分析器**

它不适合单独承担：

1. 地图和碰撞的可视化判断
2. 动画/战斗手感的最终判断
3. UI 布局的直觉性验证
4. 多轮并行开发中的唯一项目记忆

所以最成熟的方式不是“只靠 CLI 写”，而是：

- **用可视化工具看场景与 UI**
- **用自动化脚本守住回归**
- **用 AI 做编码和检查**
