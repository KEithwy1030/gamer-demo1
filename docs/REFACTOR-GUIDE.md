## 第一部分 · 产品铁则（任何 agent 干活前必读）

### 1.1 这是一款什么游戏

**流荒之路**是一款 8-15 分钟一局的**硬核轻量化搜打撤**。**2.5D 俯视角**，玩家是中世纪末世的拾荒者，钻进废弃战场搜刮装备和珍宝，撤回黑市变现。死了，全丢（除非装备在保险箱里）。

**核心循环**：搜 → 打 → 撤 → 卖。

**核心情绪**（按重要度）：
1. 搬砖快感（撤离 → 变现 → 资产积累）
2. 紧张（迷雾压力 + 玩家遭遇战）
3. 贪婪（再搜一件、再杀一只、再走 30 秒就撤）

**塔科夫式持久化哲学**：玩家档案永久（背包等级、商人信誉、累计金币等），只有当局带进去 / 当局搜到 / 当局没放进保险箱的装备会丢。

### 1.2 三条不可妥协的产品铁则

#### 铁则 A · 拾荒者主题：**装备就是力量**

玩家是普通人，能力来自捡到的装备。

- **不允许**：独立的天赋树、技能树、玩家等级、永久成长系统
- **被动加成必须依附在装备上**：回血加快、CD 缩短、搜集变快等被动效果**全部作为装备词条存在**。脱了装备，加成消失。
- 武器主动技能跟随武器（剑/刀/枪 各自 3 个），换武器就换技能
- **允许永久积累的只有**：背包容量、商人信誉、金币、保险箱里的装备——这些是"搬砖成果"，不是"战斗力增长"

#### 铁则 B · 硬核 2.5D：**不允许操作完全压过装备**

这游戏是 2.5D 俯视角、冷兵器为主、不允许"飞天遁地"。装备好的玩家在大多数情况下应该赢。

任何新机制（特别是战斗深度系统）必须遵守以下 5 条：

1. **不能让操作完全压过装备**。装备好的玩家应该是大多数对抗的胜者。
2. **禁止"高速操作秀"型机制**。包括：连续闪避连招、瞬移突进、高速位移叠加、操作越快越占便宜的设计。
3. **禁止依赖第一/第三人称物理表现的机制**。包括：背刺判定（俯视角下背向只是按键切换，太容易作弊）、弱点部位（俯视角下没有自然视觉锚点）。
4. **允许的不确定性类型**：
   - **特殊武器机制**（如杀神武器：见战斗深度系统板块）
   - **状态拖延**（流血叠加致死）
   - **环境利用**（推河 / 引怪 / 利用迷雾）
5. **不确定性是"小概率高回报"，不是"主流打法"**。新人偶尔能反杀大佬，但常规对抗依然是装备分高低。
6. **每个新机制都要回答**："这给装备好的玩家有什么影响？如果装备差的玩家用了这个机制后变成可以主动反杀大佬的常规打法，否决。"

> **当前已知调优需求**：剑的突进斩（96px 突进 + 18 伤害）速度偏快、范围偏大。重构完成后第一批数值调优要降低这个突进距离/速度。

#### 铁则 C · 板块自治：**改一个功能只动一个板块**

代码按"功能板块"切。每个板块自治，板块之间通过**广播台**通信，不直接互相调用。

- **改音效** → 只动音效相关的板块
- **改撤离** → 只动撤离板块
- **改怪物 AI** → 只动怪物板块
- **加新功能** → 新建板块或扩展现有板块，**不允许**把新功能塞进已有的总入口/主场景

agent 如果发现一个需求需要跨多个板块改动，**先停下来**，检查是不是事件清单缺了一个事件——通常加一个事件比跨板块塞代码更对。

### 1.3 工程纪律

1. **不打补丁**。bug 修在结构错误的根上，不在症状上贴胶布。
2. **新功能 = 加事件 + 加板块**，不是塞进老板块。
3. **数值变更只动板块的常量文件**，不动逻辑代码。
4. **不在客户端推断服务端没说的事**。服务端是真相源，客户端只展示。
5. **每个板块有自己的数据，不跨板块共享内部状态**。要共享数据走广播台。

1. **负责什么**：这个板块管的事

### 2.2 板块身份证（详细）

#### 板块 1 · 战斗系统

**位置**：`server/src/combat/` + `client/src/features/combat/`

**负责**：
- 玩家普攻、技能、闪避的判定
- 伤害计算（基础伤害 → 装备加成 → 暴击 → 减伤 → 最终伤害）
- 玩家死亡判定
- 攻击范围、攻击间隔、技能 CD

**不负责**：
- 怪物对玩家的攻击（在怪物系统里）
- 装备词条加成的计算（在装备系统里）
- 状态效果的持续逻辑（在状态效果系统里）
- 战斗特殊机制如杀神武器、流血叠加（在战斗深度系统里）

**发出事件**：
- `PlayerAttacked`（玩家挥出一次攻击）
- `PlayerSkillCast`（玩家释放主动技能）
- `PlayerDodged`（玩家闪避）
- `PlayerDamaged`（任何来源对玩家造成伤害——这是统一事件，环境/怪物/玩家都用）
- `PlayerCriticalHit`（暴击发生）
- `PlayerDied`（玩家死亡）

**监听事件**：
- `ItemEquipped` / `ItemUnequipped`（重新计算玩家最终属性）
- `PlayerStatsRecomputed`（玩家最终属性变化）
- `StatusEffectApplied` / `StatusEffectExpired`（调整伤害和移速）
- `EnvironmentDamageDealt`（转化为 `PlayerDamaged`）

**数据**：每个玩家的战斗运行时状态（hp、cooldown、最后攻击时间、buff 列表）

#### 板块 2 · 战斗深度系统 ⭐ 新增

**位置**：`server/src/combat-depth/` + `client/src/features/combat-depth/`

**负责**：
- **杀神武器机制**（详细规格见下）
- 流血叠加致死机制（多刀流血叠加可拖死高装备玩家）
- 环境击杀检测（推入河流 / 引怪 → 怪物击杀 / 利用迷雾扣血致死）
- 未来扩展：任何符合铁则 B 的"小概率高回报"机制

**不负责**：
- 任何让操作完全压过装备的机制（违反铁则 B）
- 任何高速位移类技能
- 背刺判定 / 弱点部位（俯视角 2.5D 不适合，已永久排除）

#### 杀神武器机制（详细规格）

**设计意图**：给战斗增加"不稳定因素"。装备好的玩家在大多数情况下依然赢，但偶尔被装备差的玩家用杀神武器反杀——这种小概率事件就是不确定性的来源。**不是鼓励新人主动挑战大佬，是给新人在被遭遇时多一次机会。**

| 项目 | 设定 |
|---|---|
| 武器基础属性 | 与同等级普通武器接近，无明显优势 |
| 触发方式 | **主动技能**（不是被动暴击） |
| 技能效果 | 高概率暴击（数值待测） |
| 技能 CD | **较长**（建议起步 45-60 秒） |
| 装备约束 | **装备时强制锁死所有其他装备槽**——视觉上玩家"光着身子"使用 |
| 对怪物 | **无效**（伤害正常或更低，防止 PvE 滥用） |
| 对玩家 | 仅对玩家有暴击特效 |
| 掉落 | 低概率但不极低（精英怪 / 富宝箱概率掉） |
| 黑市交易 | **可交易** |
| 群伤 | ❌ 无（不鼓励多人战场屠杀） |
| 蓄力 / 告警 | 技能需要 1-2 秒蓄力 + 视觉/音效告警（给目标反应时间） |

**数值平衡铁则**：
- 杀神武器使用者 vs 满装备老玩家：**胜率上限 35-40%**
- 暴击概率与暴击伤害**联动调整**，不让"一击毙命"成为常态
- 任何调整必须保证：**装备依然是常规胜利的决定因素**

- 技能 CD 精确秒数
- 掉落具体概率
- 蓄力时长 + 告警表现形式

**发出事件**：
- `SlayerSkillCharging`（杀神技能蓄力开始 → VFX/音效告警）
- `SlayerSkillTriggered`（杀神技能落地 → 暴击判定）
- `BleedStackedToLethal`（流血叠加致死）
- `EnvironmentKillRegistered`（环境击杀记录）

**监听事件**：
- `PlayerAttacked`（流血叠加判定）
- `PlayerDamaged`（叠加层数计算）
- `ItemEquipped`（如果装备的是杀神武器，记录状态）

**数据**：每个玩家的流血层数、杀神技能 CD 状态

#### 板块 3 · 状态效果系统

**位置**：`server/src/status-effects/` + `client/src/features/status-effects/`

**负责**：
- 6 种基础状态（减速 / 流血 / 减伤 / 加攻 / 加攻速 / 加移速）的施加、持续、到期
- 状态效果的层数管理（如流血叠加）
- 状态效果的来源跟踪（哪个装备 / 技能 / 消耗品 来的）

**不负责**：
- 状态效果造成的实际伤害（这是战斗系统的事；状态效果只声明"流血每秒掉 2 血"，战斗系统执行扣血）
- 流血叠加致死的判定（在战斗深度系统里）

**发出事件**：
- `StatusEffectApplied`（状态被附加）
- `StatusEffectExpired`（状态到期）

**监听事件**：
- `PlayerAttacked`（武器附带的状态效果，如刀的流血）
- `PlayerSkillCast`（技能附带的状态效果，如剑的连斩加攻速）
- `ItemUsed`（消耗品施加的状态，如尸毒抗性药）
- `ItemEquipped`（装备词条产生的常驻状态）

**数据**：每个玩家的活跃状态列表

#### 板块 4 · 怪物系统

**位置**：`server/src/monsters/` + `client/src/features/monsters/`

**负责**：
- 怪物生成（出生、节奏、位置选择）
- 怪物 AI（巡逻、追击、攻击、技能、撤退）
- 怪物攻击玩家的伤害计算
- 怪物的蓄力 / 暴怒等行为状态
- 怪物的投射物（弓箭怪的箭）

**不负责**：
- 怪物掉落物的生成（在背包系统里，监听 `MonsterKilled` 决定掉什么）
- 玩家攻击怪物的伤害计算（在战斗系统里）

**发出事件**：
- `MonsterSpawned`（怪物出生）
- `MonsterWindupStarted`（怪物开始蓄力——重构后由服务端明确广播，不再客户端推断）
- `MonsterAttacked`（怪物攻击落地）
- `MonsterEnragedStarted`（怪物进入暴怒/愤怒）
- `MonsterKilled`（怪物死亡）
- `MonsterProjectileSpawned` / `MonsterProjectileHit` / `MonsterProjectileDespawned`
- `PhaseStarted`（关卡阶段切换）

**监听事件**：
- `PlayerDamaged`（仇恨判定：是谁打的我）
- `ChestRummageStarted`（contested 宝箱噪音吸怪）
- `PlayerDied`（清除仇恨目标）

**数据**：怪物列表（runtime）、spawn 定义、阶段状态

#### 板块 5 · 宝箱系统

**位置**：`server/src/chests/` + `client/src/features/chests/`

**负责**：
- 宝箱出生位置和分布（按分区地图的规则分布）
- 宝箱状态（关闭 / 翻找中 / 已开 / 被中断）
- 翻找进度和节奏（每 1.2 秒掉一件物品）
- 宝箱掉落物的随机内容（按掉落表）
- **掉落表的时间梯度**：游戏后期掉落质量提升（激励玩家在尸毒反噬期多停留）
- 宝箱噪音机制（contested 宝箱招怪）

**不负责**：
- 玩家拾取宝箱掉落物的入包逻辑（背包系统里）
- 宝箱噪音吸引怪物的怪物 AI 反应（怪物系统里监听 `ChestRummageStarted`）

**发出事件**：
- `ChestRummageStarted`（玩家开始翻一个宝箱）
- `ChestRummageTicked`（每次翻出一件物品）
- `ChestRummageInterrupted`（被打断——受伤 / 离开范围 / 死亡）
- `ChestOpened`（宝箱被完全翻完）

**监听事件**：
- `PlayerDamaged`（如果伤害对象正在翻找，打断翻找）
- `PlayerDied`（同上）
- `PhaseStarted`（根据阶段调整掉落表权重——后期更好）

**数据**：宝箱列表、每个宝箱的状态、当前翻找者、当前对局的阶段掉落权重

#### 板块 6 · 撤离系统

**位置**：`server/src/extract/` + `client/src/features/extract/`

**负责**：
- 撤离点的位置（每张地图 **2-3 个**，分布在中央区域）
- 撤离点的开放时机（第 8 分钟）
- **火把点燃机制**：玩家必须找到火把 → 带回某个撤离点 → 点燃 → 该撤离点才能使用
- 玩家撤离读条（5 秒）
- **撤离读条期间玩家可在撤离区内移动**（不必站立不动），但离开撤离区会打断
- 撤离压力机制（点燃的撤离点对所有玩家可见，会吸引争夺）
- 撤离成功的判定和结算触发

**不负责**：
- 撤离结算（在结算系统里，监听 `ExtractSucceeded`）
- 撤离后的物品保留逻辑（背包系统 + 消耗品/工具系统的保险箱）
- 火把本身的物品逻辑（在消耗品 / 工具系统里）

**发出事件**：
- `BeaconLit`（撤离点被点燃 → 撤离区域开放 + 对其他玩家可见）
- `ExtractOpened`（撤离区域开放）
- `ExtractChannelStarted`（玩家开始读条）
- `ExtractChannelTicked`（读条进展）
- `ExtractChannelInterrupted`（被打断——离开撤离区 / 受伤 / 死亡）
- `ExtractSucceeded`（撤离成功）

**监听事件**：
- `BeaconLit`（火把点燃事件，来自消耗品/工具系统）
- `PlayerDamaged`（受伤打断读条）
- `PlayerDied`（死亡打断读条）

**数据**：撤离区状态（每个撤离点是否点燃）、每个玩家的撤离进度、压力状态

**当前测试便利性**：火把目前自动装备到玩家背包，未来调整为地图掉落物。

#### 板块 7 · 环境系统 ⭐ 新独立

**位置**：`server/src/environment/` + `client/src/features/environment/`

**负责**：
- 尸毒迷雾的阶段切换（蔓延期 / 反噬期 / 加剧期）

- 未来的其他环境危险（毒区、火区等）

**不负责**：
- 视野收窄效果（这是客户端视野系统的事，监听 `FogPhaseChanged`）

**当前 GDD 数值（待调）**：
- 反噬期 8 分钟开始（1 hp/秒）
- 加剧期 12 分钟开始（5 hp/秒）
- 总持续 15 分钟

**调整方向（重构后第一批数值调优）**：
- **反噬开始时间提前**（让玩家更早感受到压力）
- **总持续时间延长**（让"扣毒阶段"有意义，不是"必撤离"）
- **扣毒阶段允许探索**：玩家可以用药品/绷带维持 hp 继续搜刮
- **配合宝箱系统**：扣毒阶段宝箱掉落更好，激励玩家承担风险多留

**发出事件**：
- `EnvironmentDamageDealt`（环境造成伤害——会被战斗系统转化为 `PlayerDamaged`）
- `FogPhaseChanged`（迷雾阶段切换）

**监听事件**：
- 几乎不监听，自己跑 tick

**数据**：当前迷雾阶段、河流位置（从分区地图读）

#### 板块 9 · 背包系统（含永久升级）

**位置**：`server/src/inventory/` + `client/src/features/inventory/`

**负责**：

- 物品形状（剑 1×3、枪 1×4 等）和放置校验

**不负责**：

- 消耗品的实际效果（在消耗品系统里）

#### 背包永久升级机制（塔科夫式持久化）

**核心哲学**：玩家档案永久积累，不破坏"装备至上"原则——背包大不代表战斗力强，只代表能带更多东西回家。

| 项目 | 设定 |
|---|---|
| 初始容量 | 10×6（60 格） |
| 升级方式 | 玩家档案的属性（累计行为驱动） |
| 升级触发 | **撤离次数累计** + **带出珍品累计价值**（两条并行路径，具体阈值待定） |
| 死亡影响 | 升级后的容量**永久保留**，不因死亡丢失（塔科夫式） |
| 容量上限 | 待定（建议 10×10 顶配，避免过度膨胀） |

**发出事件**：
- `LootSpawned`（地面出现一个掉落物）
- `LootPickedUp`（玩家捡到东西）
- `ItemDropped`（玩家主动丢弃）
- `InventoryChanged`（背包内容变化，整体重新同步）
- `BackpackCapacityUpgraded`（背包容量永久升级）

**监听事件**：
- `MonsterKilled`（按掉落表生成怪物掉落物）
- `ChestRummageTicked`（按宝箱掉落表生成宝箱掉落物）
- `PlayerDied`（死亡掉落——除保险箱内物品）
- `ExtractSucceeded`（累计撤离次数，可能触发升级）

**数据**：每个玩家的背包格子状态、地面掉落物列表

#### 板块 10 · 装备 + 属性计算 + 装备被动 ⭐ 新独立

**位置**：`server/src/equipment/` + `client/src/features/equipment/`

**负责**：
- 装备穿戴和卸下
- 5 个装备槽（武器 / 头 / 胸 / 手 / 鞋）
- 装备品质（白 / 绿 / 蓝 / 紫 / 金）

- **特殊装备类型**：包括杀神武器等有特殊约束的装备（装备杀神武器时锁死其他装备槽）

**不负责**：

**特殊装备分类**：

#### 保险箱机制（局内兜底）

**核心定位**：保险箱是**局内特殊容器物品**，不是局外大厅仓库。

| 项目 | 设定 |
|---|---|
| 类型 | 玩家带入对局的特殊物品 |
| 占位 | 占据玩家背包格子（具体格数待定） |
| 功能 | 玩家局内可将珍贵装备/珍品放入保险箱 |
| 死亡影响 | **保险箱内的所有物品在玩家死亡时保留**，跟随玩家回到大厅 |
| 容量 | 有限（具体待定，建议 2×3 = 6 格） |
| 获取 | 从大厅带入对局（每局一个） |

**这等于是"局内的小型仓库"**——给玩家"如果搜到了真的好东西，可以兜底"的安全感，鼓励冒险搜刮。

#### 火把机制

| 项目 | 设定 |
|---|---|
| 当前测试态 | 开局自动装备到玩家背包 |
| 未来实现 | 地图特定位置生成，玩家需要找到并带回撤离点 |
| 使用方式 | 在撤离点激活使用 → 点燃撤离点 → 触发 `BeaconLit` 事件 |
| 数量 | 每张地图 2-3 个火把（对应 2-3 个撤离点） |

**发出事件**：
- `ItemUsed`（玩家使用一件消耗品/工具）
- `BeaconLit`（火把点燃归营火 → 撤离系统监听）
- `ItemDepositedInSafe`（物品放入保险箱）
- `ItemWithdrawnFromSafe`（物品从保险箱取出）

**监听事件**：
- `PlayerDied`（处理保险箱内物品的保留逻辑）
- `ExtractSucceeded`（保险箱内物品和背包内物品一起带出）

**数据**：消耗品定义、活跃工具状态、每个玩家保险箱内的物品列表

## 第四部分 · 广播台是什么、怎么用

### 4.1 类比理解

广播台像一个**电台**。

- **任何板块都可以拿起话筒喊话**：例如怪物板块喊"3 号怪物死了，掉在 (200, 300)"
- **任何板块都可以打开收音机听**：战斗特效板块听到"3 号怪物死了" → 在 (200, 300) 播一个爆裂动画；HUD 板块听到 → 显示击杀提示；音乐系统听到 → 决定是否切换 BGM
- **喊话的板块不知道谁在听**，听的板块也不知道是谁喊的。它们只和广播台对话。

### 4.2 服务端的广播台

每个**房间**有自己的广播台（互不干扰）。

```
房间 ABC123:
  广播台队列: [事件1, 事件2, 事件3, ...]
```

服务端跑一个 tick（每秒 10-20 次），各板块往队列里塞事件。tick 末尾**一次性**把队列里的事件通过 socket 发给所有客户端，然后清空队列。

**关键纪律**：板块永远不直接调用其他板块。要影响别的板块，**塞个事件到队列**就好。

### 4.3 客户端的广播台

整个游戏一个广播台（玩家只在一个房间里）。

```
客户端广播台:
  收到 socket 推来的事件 → 喊话给所有订阅者
  本地产生的事件（输入、UI 交互） → 同样喊话
```

各个客户端板块在启动时**订阅自己关心的事件**，事件来了就响应。

### 4.4 板块之间不再有直接联系

**重构前**（错的）：
```
战斗系统 → 调用 → 撤离系统的"中断读条"方法
战斗系统 → 调用 → 宝箱系统的"中断翻找"方法
战斗系统 → 调用 → 音效模块的"播放受伤音"
```

**重构后**（对的）：
```
战斗系统 → 喊"PlayerDamaged"事件 → 广播台
                                      ↓
撤离系统听到 → 自己决定要不要中断读条
宝箱系统听到 → 自己决定要不要中断翻找
音效模块听到 → 自己决定播什么音
```

**好处**：战斗系统不知道有谁在听。明天加一个新模块（比如成就系统："累计受伤 100 次解锁成就"），新模块自己订阅 `PlayerDamaged`，**不动战斗系统一行代码**。

## 第五部分 · 施工顺序（30 小时怎么分步）

整个重构分 6 步。**每一步做完游戏都还能玩**——这样如果中途出现别的紧急需求，可以停下来再回来接。

**验收**：

- 共享层有完整事件清单，但还没人用

**状态**：游戏运行完全不变（因为没人用新东西）。

### 步骤 2 · 建立 23 个板块的目录骨架（2-3 小时）

### 步骤 3 · 服务端广播台 + 核心 4 个板块改造（8-10 小时）

5. **双轨保留**：旧的 socket event 仍然发，新的也发。客户端两边都能收到，不会突然瞎掉。

**不做什么**：客户端暂时不动（继续按旧方式吃事件）。

**验收**：
- 玩 5 分钟游戏，行为和重构前完全一致

## 第六部分 · 未来扩展指南

**这份文档的真正威力在重构完成之后**。任何新功能 / 新调整 / bug 修复都按下面的流程走：

### 6.1 加新功能

**1. 问自己**：这是 23 个板块里的哪一个？

- 在已有板块里 → 直接进那个板块开干
- 不在任何板块里 → 看看是不是预留板块 → 是的话，激活它（按身份证模板写实现）
- 全新功能不属于任何板块 → 新建板块 + 在这份文档里加身份证 + 更新事件清单

**2. 问自己**：这个新功能需要广播什么事件？

- 列出该板块会发出的新事件
- 列出该板块会监听的事件
- 在共享层事件清单文件里添加新事件格式

**3. 实现**

- 只在该板块文件夹里写代码
- **不要碰其他板块**
- 如果发现需要碰其他板块——停下来，问自己是不是事件清单缺了一个事件

### 6.2 调整数值

数值放在各板块的 `constants.ts` 文件里。**调数值只动这个文件，不动逻辑**。

例如要调"剑突进的距离从 96 → 60"：
- 进 `server/src/combat/constants.ts`
- 改一行
- 完事

不动战斗逻辑、不动 client 任何东西。

### 6.3 修 bug

**1. 通过事件找板块**：bug 的现象是什么？

- 音效不对 → 看是哪个事件触发的音效 → 进客户端对应板块
- 视觉特效不对 → 同上
- 服务端逻辑错 → 看是哪个事件发出的不对 → 进服务端对应板块

**2. 只动一个板块**

如果发现"修这个 bug 需要动 3 个板块"，**停下来**。99% 的情况是你修在了错的层——真正的问题可能在更上游的板块没正确发事件。

### 6.4 加预留板块的真实实现

当某个预留板块（如"打造/强化"）要真正开发时：

1. 找到该板块的身份证（在这份文档里）
2. 按身份证的"发出事件"清单，在共享层事件清单文件加这些事件
3. 在板块文件夹里开始写实现
4. 监听的事件按身份证订阅
5. 更新这份文档：把状态从"预留"改为"已有"

**整个过程不动其他任何板块**。

### 6.5 agent 工作前的 checklist

任何 agent 在接到任务后，按下面顺序检查：

1. ☐ 我读过第一部分的产品铁则吗？
2. ☐ 这个任务属于哪个板块？
3. ☐ 我要发的事件 / 监听的事件，在事件清单里有吗？
4. ☐ 我会动几个板块？如果超过 1 个，是不是事件清单缺事件？
5. ☐ 我会写新的"客户端推断服务端状态"吗？（如果会 → 停下，把这件事变成服务端事件）
6. ☐ 数值改动放在 `constants.ts` 而不是逻辑代码里？
