# Playtest Retrospective · 2026-05-06

本记录用于给后续 CodexAgent / 其它 AI 接手时避坑。它不是任务清单，也不是修复方案；它记录本轮长时间执行后，用户实际体验与 agent 自动验收之间出现的偏差。

## 背景

本轮执行持续约 5-6 小时，连续提交了锁定辅助、Boss、湿毒河、战斗可读性、队伍火种撤离、拖拽 contract 验证等内容。自动验证最终全部为绿，但用户进入实际画面后反馈：很多功能在代码和脚本里存在，却没有在游戏画面和操作手感中有效体现。

核心教训：自动验证只能证明 contract 存在，不能替代视觉、手感、资产、地图读感和玩法可理解性的真实验收。

## 用户反馈核查结论

### 1. 自动锁定 / 后撤

结论：部分成立。

代码中存在锁定辅助、短追击、进入合法距离后出手、后撤输入取消追击等逻辑，相关文件包括：

- `client/src/scenes/gameScene/lockAssist.ts`
- `client/src/scenes/GameScene.ts`
- `client/src/scenes/gameScene/inputBridge.ts`
- `scripts/validate-lock-assist.ts`

但当前没有明确的前端锁定 UI、目标框、锁定提示、追击状态提示或后撤取消反馈。脚本只能证明规则存在，不能证明玩家能感受到“自动锁定”和“后撤脱管”。

后续要求：凡是宣称“锁定”“追击”“后撤”的功能，必须同时考虑玩家可见反馈和真实输入手感，不能只用 helper contract 当完成证据。

### 2. Boss / 精英 / 普通怪外观

结论：用户反馈成立。

当前资源目录只看到普通怪和精英怪两套图：

- `monster_normal_sheet_4x4.png`
- `monster_elite_sheet_4x4.png`

Boss 复用了精英怪底图，只靠尺寸、颜色、光环、血条、文本标签区分。没有 Boss 独立形象，也没有 Boss 专属动作分帧。

后续要求：如果需求说 Boss 应该是独立怪物，不能用“精英怪放大 + 改色 + BOSS 标签”交付。必须有独立资产和动作帧，至少覆盖 idle / move / attack / skill / hurt / death 中的关键状态。

### 3. Boss 技能反馈

结论：用户反馈成立。

当前 `charge / smash / enrage / recover` 的前端表现主要是：

- 文本 label 改变
- phase bar
- telegraph ring 缩放或变色
- sprite tint / alpha / angle / scale 的轻微变化

没有专门的冲锋动画、重击动作、狂暴变身动作、位移残影或技能分帧。玩家从画面上很难理解“它正在冲锋 / 重击 / 狂暴”，只能看到数值或状态在后台发生。

后续要求：战斗技能不能只通过状态字段、速度变化、环形占位符表达。冲锋应有位移动作或路径预警，重击应有明显前摇和打击动作，狂暴应有持续可辨识状态变化。

### 4. 普通怪 / 精英怪大小变化

结论：用户反馈成立。

当前 `MonsterMarker` 中存在三级尺寸：

- 普通怪：`120`
- 精英怪：`158`
- Boss：`212`

这意味着普通怪和精英怪也被明显改大了，而用户本意是只需要 Boss 看起来更大，普通怪和精英怪不应因为本轮 Boss 工作被顺手扩大。

后续要求：不要把“让 Boss 更突出”扩展成“所有怪物尺寸重做”。未被要求的视觉比例调整必须谨慎，尤其是会改变已有画面观感的内容。

### 5. 湿毒河 / 撤离点

结论：部分成立，且风险较高。

静态坐标显示撤离点位于地图中心附近，湿毒河段 `river_extract_weir` 靠近中心撤离点。服务端 hazard 判定只排除了 safe crossing，没有看到对 extract zone 的专门豁免。客户端绘制也直接画 river rect，再画撤离圈和 beacon。

这说明即使自动验证能证明“关键点不在 hazard 中心”，实际视觉上仍可能出现河流压近或覆盖撤离点的读感问题。

后续要求：地图 hazard 不能只验证点位中心。必须同时验证：

- 撤离圈整个半径范围是否安全
- hazard 视觉是否覆盖撤离 UI
- safe crossing 和视觉桥/浅滩是否真正贴合
- 玩家真实路线是否可读

### 6. 局内背包和大厅仓库拖拽

结论：部分成立。

局内和大厅当前确实是两套前端实现：

- 局内：`client/src/ui/InventoryPanel.ts`
- 大厅：`client/src/ui/stashView.ts`

两边后端 contract 都支持 move / swap，也都有自动验证，但前端结构、区域模型、拖拽反馈、动作菜单和交互节奏不同。大厅仓库拖拽体验相对完整，局内背包更轻量，用户感受到“不够丝滑”和“不统一”是合理的。

后续要求：用户要求“背包仓库整理逻辑是一套”时，不能只统一后端 move contract。前端交互、拖拽 ghost、落点高亮、非法反馈、装备位互换、格子吸附节奏也应统一。

## 这次执行的主要失误

1. 把自动验证全绿误判成用户体验达标。
2. 把“代码有状态”误判成“玩家能看见功能”。
3. 用颜色、大小、标签替代了 Boss 独立资产需求。
4. 让普通怪 / 精英怪也跟着 Boss 工作被改大，超出了用户目标。
5. 地图验证只看点和矩形 contract，没有充分检查视觉覆盖和路线读感。
6. 背包和仓库只统一了部分数据 contract，没有统一前端拖拽体验。
7. 对“人工验证项”的记录不够强硬：应明确告诉用户哪些必须实际进游戏看，而不是用脚本结论包装成完成。
8. 长时间执行时，主控层应周期性要求浏览器截图或画面证据，而不是只看 subagent 的命令绿灯。

## 后续 agent 必须遵守的验收原则

1. 凡是涉及视觉、战斗技能、地图、UI 拖拽、操作手感，都必须拿实际画面或可视证据验收。
2. Boss 相关任务默认需要独立资产和动作帧；如果没有资产，就只能称为“Boss 逻辑骨架”，不能称为完整 Boss。
3. 地图 hazard 必须同时验服务端判定和客户端画面，尤其是撤离点、桥、浅滩、资源点附近。
4. “可读性”不等于加文字、加环、改颜色。玩家必须能从动作和画面变化理解事件。
5. 背包 / 仓库 / 装备栏若被要求同一整理逻辑，应优先抽象统一交互模型，而不是分别在两个 UI 文件中各写一套拖拽。
6. 对用户明确没有要求的视觉比例、尺寸、样貌，不要顺手改。
7. 自动验证脚本应标明覆盖边界，尤其要区分 contract 验证和真实体验验证。
8. 每个长 loop 结束前，主控层必须做“玩家视角复查”：这个功能用户能不能看见、能不能理解、能不能感受到。

## 当前已确认但未修复的问题

本文件只记录问题，不包含修复。

- Boss 没有独立形象和动作分帧。
- Boss 技能前端反馈不足，冲锋 / 重击 / 狂暴主要是状态和简单图形变化。
- 精英怪也被明显放大，超出“只让 Boss 大”的期望。
- 湿毒河视觉和撤离点关系存在高风险，需要重新看画面和坐标设计。
- 局内背包和大厅仓库拖拽是两套前端体验，仍需统一。
- 锁定 / 后撤虽然有逻辑和脚本，但缺少明确可见反馈和真实手感验收。
