# Agent Release Lessons · 2026-05-06

本文件给后续接手 `E:\CursorData\gamer` 的 coordinator、developer worker、tester worker 使用。它记录这两轮 Goal 后暴露出的执行问题，不是进度清单。

## 这次为什么会留下低级尾巴

核心原因不是单个 bug 难，而是验收标准错位：

1. 主调度层把“自动脚本通过”误判为“玩家可上线体验通过”。
2. developer worker 倾向于补 contract、状态字段、快捷测试入口，但没有被强制证明玩家实际看见和操作到了这些功能。
3. tester worker 的验证偏向“能进入、能截图、无 CORS”，没有按玩家反馈做反向挑刺。
4. 浏览器验收只证明了场景存在，没有证明关键交互完成，例如锁定后是否真的能稳定命中、撤离读条是否稳定、黑市交互是否低门槛。
5. 主调度层没有把用户一开始强调的“可上线、不能半成品”翻译成可执行的放行门槛，而是接受了过宽的证据。

结论：以后不能再把构建绿、脚本绿、截图存在当成 release 绿。游戏功能必须多一层“玩家视角门槛”。

## Subagent 分工必须改变

### Coordinator

主会话只做：

- 把用户反馈拆成可验收任务包。
- 定义每个任务的“不通过条件”。
- 审查 worker 证据是否覆盖用户原话。
- 拒绝只给脚本结果、没有玩家视角证据的交付。

主会话不应该长时间亲自改代码；如果必须临时改，也要转回 worker 复核。

### Developer worker

开发 worker 每个任务必须交付：

- 改了哪些文件。
- 改动怎样对应用户反馈。
- 哪些自动测试覆盖了逻辑。
- 哪些浏览器证据覆盖了画面或交互。

涉及 UI、手感、比例、战斗可读性的任务，不允许只交 contract test。

### Tester worker

测试 worker 必须站在用户角度挑错，而不是只证明开发说得对。

测试报告必须包含：

- 用户原话对应的验收项。
- 实际复现步骤。
- 通过证据和失败证据。
- 是否存在“看起来有功能，但玩家用起来不成立”的情况。
- 是否需要人工手感确认。

tester worker 不能把“打开页面成功”“截图存在”“console 没红”当作充分验收。

## 下一轮必须修的用户反馈

### 1. 自动锁定和攻击判定

用户反馈：偶尔有自动锁定弹窗，但人物攻击判定和实现没有达到可上线。

下轮要求：

- 锁定不能只是提示弹窗，必须让攻击行为稳定受益。
- 目标选择、距离判定、追击进入攻击距离、攻击出手、取消锁定、后撤取消都要连起来。
- 必须有玩家可见反馈：锁定目标框、目标切换、锁定丢失、追击/取消状态。
- 验收不能只看 `validate-lock-assist`，必须进浏览器实际打怪并核对命中、未命中、取消。

### 2. Web 画面比例和资产粗糙

用户反馈：画面比例还是不对，这版更像简单放大；资产质量高，但放进游戏显粗糙。

下轮要求：

- 检查 canvas 尺寸、CSS 缩放、Phaser camera、sprite display size、pixelArt/antialias 设置。
- 不能用简单整体放大解决适配问题。
- 必须同时截桌面常用窗口和至少一个窄屏窗口，检查 UI、角色、地图纹理、怪物边缘。
- 视觉验收要看“资产在游戏中是否清晰”，不是只看素材源文件质量。

### 3. 黑市界面复用仓库格子交互

用户反馈：黑市不应使用列表显示自己的装备，应复用仓库格子界面，降低交互门槛。

下轮要求：

- 黑市里的“我的装备/可挂售物品”必须用格子展示，而不是列表。
- 尽量复用仓库 grid、装备位、拖拽/选中/高亮/非法反馈模式。
- 挂单逻辑可以保留，但入口交互要像仓库：从格子里选物品，再填写价格/确认挂售。
- 验收必须截图黑市格子化界面，并实际完成一次挂单/取消。

### 4. 玩家、Bot、普通怪、精英怪、Boss 比例体系

用户反馈：普通怪也太大，怪物大小没有层级，每个怪看起来差不多。

下轮要求：

- 集中定义比例表：玩家、队友 Bot、普通怪、精英怪、Boss。
- 普通怪必须明显小于精英怪，精英怪小于 Boss。
- Boss 可以有压迫感，但不能靠把所有怪都做大来解决。
- 验收必须同屏截图普通怪、精英怪、Boss 或分别给出可比较截图。

### 5. 撤离点进度条闪现

用户反馈：进入撤离点时进度条闪现；之前稳定读条，可能是 HUD 或判定有断点。

下轮要求：

- 检查服务端 extract zone 判定、player position sync、HUD 状态更新和 interrupted/started 来回切换。
- 需要防止边界抖动：必要时加入半径容差、状态保持或更稳定的读条条件。
- 验收必须实际站在撤离点内持续读条，确认进度条不闪、不反复重置。

### 6. 多人联机

用户反馈：这一阶段暂时没有两人同时跑一局条件，等下次强调后再集中测。

下轮处理：

- 记录为待办，不把“未测多人真实联机”包装成已上线。
- 可以保留 socket 双客户端自动化，但不能宣称真实朋友跨机器已验证。

## 新的放行门槛

任何下轮 Goal 结束前，必须按以下层级审计：

1. 自动化：`npm run build`、相关 contract test、必要的主线脚本通过。
2. 浏览器：每个用户反馈项都有真实浏览器截图或可观察证据。
3. 交互：锁定、攻击、撤离、黑市挂售必须至少完成一次真实操作链，不只截图静态画面。
4. 视觉：比例、清晰度、怪物层级必须由截图证明。
5. 人工项：如果必须真人手感判断，必须明确列入“待人工验证”，不能写成已完成。

## 下轮 worker 提示词要求

给 developer worker 的提示词必须包含：

- 不要只修脚本，不要只补状态字段。
- 每项修复要交付玩家可见证据。
- 不要扩大范围改未要求的视觉比例。
- 不要把 dev preset 当正式体验。

给 tester worker 的提示词必须包含：

- 先复述用户原话，再逐项验收。
- 主动寻找“功能看起来存在但用起来不成立”的问题。
- 截图必须覆盖目标交互完成后的状态。
- 报告必须区分自动化通过、浏览器通过、人工待验。
